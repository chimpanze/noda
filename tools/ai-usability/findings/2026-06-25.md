## Fix response.json/response.error status field type in node schema (declared string, examples use integer)

- **Labels:** type:documentation, component:plugin, type:feature
- **Hit by:** 02-basic, 05-hard  (impact: 2)

### Context
The `response.json` and `response.error` node schemas are the authoritative type source a builder consults via `noda_get_node_schema` to learn how to configure a node.

### What is missing or confusing
Both `noda_get_node_schema(response.json)` and `noda_get_node_schema(response.error)` declare `status` as `{"type":"string","description":"HTTP status code"}`. Yet every MCP-reachable example uses a bare integer literal: `noda_get_examples(crud)` uses `"status": 201` / `400`; the scaffolded `workflows/hello.json` uses `"status": 200`; `noda://docs/quick-start` and `noda://docs/expression-cookbook` use numeric `200`/`201`/`401`/`404`. The `noda://docs/testing` Common Errors table even warns "JSON numbers are float64, so use `201` not `\"201\"`... handles int/float coercion but not string/number", implying integer is canonical and string config will break integer test assertions. `noda_validate_config` accepts BOTH forms, so the declared `type:string` is an inaccurate annotation, not an enforced constraint — and it actively misled builders into choosing strings.

### How to reproduce (briefs)
02-basic and 05-hard (reported twice in 05-hard). Call `noda_get_node_schema(response.json)` and compare `config_schema.properties.status.type` against any `noda_get_examples(crud)` workflow.

### Suggested fix
Change the `status` type in the `response.json` and `response.error` node config schemas to `integer` (or `oneOf` number/string) so the declared type agrees with every example, the scaffold, the docs, and the test runner's coercion behavior.

## Include file path and JSON pointer/field name in noda_validate_config errors

- **Labels:** enhancement, component:config, type:feature
- **Hit by:** 02-basic  (impact: 1)

### Context
`noda_validate_config` returns a rich summary object on success (`file_count`, per-category counts, `valid:true`) but a bare message array on failure.

### What is missing or confusing
A validation failure returns `{"valid":false,"errors":["got string, want object"]}` with NO file path, no JSON pointer, and no field name. The tool's input schema exposes only `config_dir` — there is no verbose/detail flag. This is genuinely ambiguous: `noda_get_config_schema(route)` shows four distinct fields typed `object` that could each emit "got string, want object" (`body.schema`, `params.schema`, `query.schema`, `response.<status>.schema`), so the message cannot be mapped to a field on its own. The needed location info is unreachable via any other `noda_*` tool or `noda://` resource. The builder had to reason it out from the route schema and re-test by trial.

### How to reproduce (briefs)
02-basic. Set `routes/api.json` `body.schema` to the string `"$ref(schemas/greeting.json)"` and call `noda_validate_config` — it returns the bare `["got string, want object"]` with no localization.

### Suggested fix
Include the offending file path and JSON pointer/field name in each validation error, e.g. `"routes/api.json: body.schema: got string, want object"`. Match the detail level of the success summary.

## Flag or document stale scaffolded tests whose asserted output no longer matches the workflow

- **Labels:** enhancement, type:test, component:config
- **Hit by:** 01-trivial  (impact: 1)

### Context
The scaffold ships `tests/hello.test.json` asserting a specific output against the original scaffolded `hello` workflow. After a builder edits the workflow to meet a brief, this assertion becomes semantically stale.

### What is missing or confusing
`noda_validate_config` still reports `valid:true` because it only checks test-file structure, not whether the asserted output matches the current workflow. `noda_explain_workflow` is explicitly documented as not executing the workflow — it exposes the terminal `response.json` body keys (allowing a manual key-level diff) but never resolves values, so it cannot catch a value mismatch and emits no warning. The mechanism that actually catches a stale test is the CLI test runner (`noda test`), but there is NO MCP tool that runs tests — the full MCP tool list is scaffold/list/read/validate_config/get_config_schema/list_nodes/get_node_schema/list_functions/validate_expression/explain_workflow/get_examples. So staleness detection is unreachable through the MCP surface.

### How to reproduce (briefs)
01-trivial. Edit the scaffolded workflow, then run `noda_validate_config` — it reports `valid:true` despite the now-stale `tests/hello.test.json` assertion.

### Suggested fix
Either have a tool (or `noda_validate_config`) statically flag tests whose asserted output keys cannot be produced by the referenced workflow, or document clearly (and ideally in the scaffold output) that scaffolded tests are illustrative and must be updated when the workflow changes.

## Document the database migration file format, naming convention, and discovery/apply behavior

- **Labels:** documentation, type:documentation, component:plugin
- **Hit by:** 03-data  (impact: 1)

### Context
Briefs that require creating a database table need a migration. `noda_scaffold_project` creates an empty `migrations/` directory.

### What is missing or confusing
`noda://docs/quick-start` does establish that `migrations/` holds "SQL migration files" (not a Noda config type), but nothing on the MCP surface documents: (a) the migration file naming convention (there is no analog to the testing doc's `test-<workflow-id>.json` rule), (b) up/down semantics, or (c) whether/how Noda discovers and applies migration files (no `noda migrate` command appears anywhere; quick-start only shows init/start/validate/test/dev). `noda_get_config_schema` has no `migration` type; `noda_scaffold_project` leaves `migrations/` empty; `noda_list_project_files` has no migrations category; `noda_validate_config`'s summary omits migrations entirely. No example ships a migration despite crud/auth writing to `users` and the quick-start tutorial writing to `tasks` — tables no example ever creates. A builder must guess a filename (e.g. `0001_create_notes.up.sql`/`.down.sql`) whose discovery/apply behavior is unverifiable through MCP.

### How to reproduce (briefs)
03-data. Call `noda_scaffold_project`, then try to find any schema, example, or doc describing the migration file format — `noda_get_config_schema` has no `migration` type and no example contains a migration file.

### Suggested fix
Add a `migration` config_type (or doc resource) documenting the file format and naming convention, include a migration file in `noda_get_examples` (especially `crud`), have `noda_scaffold_project` drop a sample migration into `migrations/`, and have `noda_validate_config` recognize and report migrations.

## Reconcile route-file array form: Authentication Guide shows arrays the validator rejects

- **Labels:** documentation, type:documentation, component:config
- **Hit by:** 04-auth  (impact: 1)

### Context
The Authentication Guide (`noda://docs/authentication`) shows route files as top-level JSON arrays of multiple route objects in two places: the "Authorization Code Flow" section (`routes/auth.json` = `[{oidc-login},{oidc-callback}]`) and the "Public and Protected Routes" common pattern.

### What is missing or confusing
The route config schema and `noda_validate_config` only accept a single route object per file. Following the guide produces the validator error `invalid JSON at offset 0: expected { character for map value`, which gives no hint that arrays are unsupported or that routes must be one-per-file. The contradiction is not resolved anywhere reachable: `noda_get_config_schema(route)` describes a single object with no array/oneOf form but never states arrays are disallowed; all examples and the quick-start consistently use one route object per file.

### How to reproduce (briefs)
04-auth. Put a top-level JSON array of route objects in `routes/combined.json` (as the auth guide shows) and run `noda_validate_config` — it returns `routes/combined.json: invalid JSON at offset 0: expected { character for map value`.

### Suggested fix
Either support a top-level array in route files (to match the docs) or update the Authentication Guide examples to show one route object per file. Also improve the validator error to say `route files must contain a single route object; arrays are not supported — split into separate files.`

## Document role-claim RBAC via casbin.enforce and expand the opaque security.casbin schema

- **Labels:** documentation, type:documentation, component:server
- **Hit by:** 04-auth  (impact: 1)

### Context
The Authentication Guide's Casbin section documents `casbin.enforce` middleware deriving the subject from `auth.sub` (user ID) with static `role_links` mapping specific user IDs to roles.

### What is missing or confusing
There is no documented way for `casbin.enforce` to consume the token's `auth.roles` claim directly, so a generic "any token carrying the admin role" check at the middleware layer is not achievable without enumerating users. The documented `security.casbin` fields (model, policies, role_links, tenant_param) offer no way to use `auth.roles` as the subject, and `noda_get_config_schema(root)` shows `security.casbin` as literally `{"type":"object"}` with zero detail. `casbin.enforce` is middleware (not a node), so `noda_get_node_schema` returns not-found and there is no middleware-schema introspection. The achievable path is an in-workflow `control.if` on `{{ 'admin' in auth.roles }}` (shown in the doc's "Role-Based Response Filtering" section but not explicitly for returning 403).

### How to reproduce (briefs)
04-auth. Read `noda://docs/authentication` Casbin section and call `noda_get_config_schema(root)` — there is no documented mechanism for role-claim-based middleware enforcement without per-user `role_links`.

### Suggested fix
Document how `casbin.enforce` can match on the token's roles claim (e.g. a config option to use `auth.roles` as subject), or add an explicit example showing role-claim-based 403 enforcement at the route/middleware layer without per-user `role_links`. Expand the `security.casbin` config schema rather than leaving it an opaque object.

## Add a realtime/connections doc covering the WebSocket/SSE subscription, channel, and lifecycle model

- **Labels:** documentation, type:documentation, component:server
- **Hit by:** 05-hard  (impact: 1)

### Context
Configuring a realtime WebSocket endpoint (cross-instance sync, channel broadcast) currently has to be reconstructed from the connections JSON schema plus a single example. There is no dedicated doc resource for realtime/WebSocket/SSE/connections.

### What is missing or confusing
Three core needs are absent from every MCP resource/tool: (a) the client subscription model — nothing describes how a client subscribes to a channel or how clients map onto `channels.pattern`; (b) the `channels.pattern` to ws.send `channel` relationship — `pattern` has no description in the connections schema, `max_per_channel` is unexplained, and nothing explains how the endpoint pattern matches/constrains the channel string passed to `ws.send` (the heart of broadcast routing); (c) the `on_connect`/`on_message`/`on_disconnect` lifecycle, including the undocumented `{{ message }}` context variable used in the example's `on_message` input (`noda://docs/data-flow` enumerates request fields but never `message`). The `sync.pubsub` requirement and the basic endpoint shape ARE reachable (connections schema + `noda://docs/services`), but the conceptual routing/subscription/lifecycle model needed to build it confidently is not.

### How to reproduce (briefs)
05-hard. Call `ListMcpResourcesTool` (8 docs, none for realtime/connections), `noda_get_config_schema(connections)`, and `noda_get_examples(websocket)` — the subscription model, channel-to-pattern relationship, and handler lifecycle are not documented anywhere.

### Suggested fix
Add a `noda://docs/realtime` (or `docs/connections`) resource covering: endpoint definition, the `sync.pubsub` requirement, the channel naming/subscription model, how `channels.pattern` relates to the `ws.send`/`sse.send` channel value, the `on_connect`/`on_message`/`on_disconnect` lifecycle with their context variables (including `message`), and a full POST -> store -> ws.send broadcast walkthrough.

## Reconcile connection handler shape: schema declares on_connect/on_message as string, example uses object

- **Labels:** documentation, type:documentation, component:config
- **Hit by:** 05-hard  (impact: 1)

### Context
Wiring `on_connect`/`on_message`/`on_disconnect` handlers on a WebSocket endpoint requires knowing their shape.

### What is missing or confusing
`noda_get_config_schema(connections)` and `noda_get_config_schema(root)` both declare `on_connect`/`on_message`/`on_disconnect` as `"type":"string"`. But `noda_get_examples(websocket)` (and the same snippet under `noda_get_examples(all)`) shows them as objects: `"on_connect": {"workflow": "ws-connect", "input": {...}}`. The two surfaces directly contradict each other, and the example additionally uses a flat `{id,type,path}` endpoint object rather than the schema-required `endpoints.<name>` map — so it cannot even be copied verbatim. No reconciling documentation exists: `noda://docs/data-flow` documents `trigger.input` only for routes and never explains how a connection handler supplied as a workflow-id string receives input, nor defines the `message`/`auth.user_id` variables the example references. The validator cannot disambiguate because the project has no connections config to validate the shapes against.

### How to reproduce (briefs)
05-hard. Compare `noda_get_config_schema(connections)` (`on_connect` = `type:string`) against `noda_get_examples(websocket)` (`on_connect` = `{workflow, input}` object) — they directly contradict, with no reconciling source.

### Suggested fix
Make the example and the schema agree. If handlers are workflow-id strings, document how trigger input is mapped for connection workflows and define the handler context variables (`message`, etc.); if they are objects, update the schema to that shape and fix the example's outer endpoint shape to match the `endpoints.<name>` map.

## Make scaffolded service shape match the documented/root-schema service shape

- **Labels:** documentation, type:documentation, component:config
- **Hit by:** 05-hard  (impact: 1)

### Context
Services are declared in `noda.json`. Builders compare the scaffold output against `noda://docs/services` and `noda_get_config_schema(root)`.

### What is missing or confusing
`noda_scaffold_project` emits services with concrete-impl plugin names and connection fields at top level: `{"db":{"plugin":"postgres","dsn":"..."}, "cache":{"plugin":"redis","url":"..."}}`. But four canonical sources unanimously prescribe a family-name plugin with nested config: `noda://docs/services` (`{"plugin":"db","config":{"url":"..."}}`), `noda://docs/quick-start`, and `noda_get_config_schema(root)` (`services.additionalProperties` has `{plugin, config}`, `required:[plugin]`). These two shapes are incompatible (family-vs-impl name; top-level-vs-nested fields). Critically, `noda_validate_config` gives no corrective signal — it reports `valid:true` for both the freshly scaffolded project AND a hand-built project using the scaffold shape — silently blessing the contradictory scaffold output.

### How to reproduce (briefs)
05-hard. Call `noda_scaffold_project` and inspect `noda.json` services, then compare against `noda://docs/services` and `noda_get_config_schema(root)` — the shapes differ and `noda_validate_config` accepts both.

### Suggested fix
Make the scaffold output match the documented/root-schema service shape (plugin family name + nested `config`), or document that both forms are accepted and how concrete plugin names map to families.

## Document the ws.send/sse.send connections service-slot binding and channel-to-pattern relationship

- **Labels:** documentation, type:documentation, component:plugin
- **Hit by:** 05-hard  (impact: 1)

### Context
`noda_get_node_schema(ws.send)` declares a required `connections` service slot. The Service Wiring Guide (`noda://docs/services`) establishes that a slot value is normally a service instance name from `noda.json` "services".

### What is missing or confusing
The services doc lists the `connections` slot as "Connection manager (built-in)" with NO instance to configure and NO entry in services — which implies no value is needed, contradicting the schema's `Required=true`. Nothing states whether `services.connections` should reference a connections *endpoint* name (the builder guessed `"board"`), be omitted, or reference something else. Separately, no reachable doc/example/schema connects the `ws.send` `channel` field to an endpoint's `channels.pattern`. The `noda_get_examples(websocket)` snippet contains no `ws.send` node and uses a non-schema endpoint shape, so it demonstrates neither the slot binding nor the channel relationship. `noda_validate_config` passes a guessed value and even reports `"connections": 0` despite a defined `connections.endpoints.board` block — so a wrong endpoint name would pass silently.

### How to reproduce (briefs)
05-hard. Call `noda_get_node_schema(ws.send)` (required `connections` slot) and read `noda://docs/services` (slot listed as built-in with no instance) — the binding value and the `channel`-to-`channels.pattern` relationship are undocumented.

### Suggested fix
Document the `connections` slot binding for `ws.send`/`sse.send`: whether/what to put in `node.services`, and how the node's `channel` value maps to an endpoint's `channels.pattern`. Have `noda_validate_config` cross-check `services.connections` against defined endpoints.
