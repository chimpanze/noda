## Fix or document $ref resolution for JSON Schema references in route.body.schema / transform.validate

- **Labels:** bug, documentation, component:config
- **Hit by:** 02-basic  (impact: 1)

## Context
The `crud` example returned by `noda_get_examples` uses `{"$ref": "schemas/User"}` inside a `transform.validate` node config, implying that JSON Schema files under a project's `schemas/` directory can be referenced by a relative path. `noda_list_project_files` also lists `schemas` as a first-class project-file category, implying $ref-based reuse is a real, intended feature.

## What is missing or confusing
No `noda_*` tool documents the $ref base-path/resolution rule (project-root-relative? a registered schemas map? file extension required or not? `$id`-based lookup?). Worse, the rule appears to not work at all through any syntax reachable via the MCP surface — including the crud example's own documented pattern applied 1:1 to a real project schema file.

## How to reproduce
Brief: `02-basic` (build phase). Scaffold a project with `schemas/greeting.json`, then set `route.body.schema` (or a `transform.validate` node config, mirroring the crud example) to `{"$ref": "schemas/greeting"}`.

Evaluator evidence: "I directly edited the project's route (routes/greet.json) and workflow (workflows/greet.json) files to try every plausible $ref syntax against noda_validate_config as the oracle -- \"schemas/greeting\", \"schemas/greeting.json\", \"./schemas/greeting.json\", and bare \"greeting\" -- both as route.body.schema and inside a transform.validate node (mirroring the crud example's exact {\"$ref\": \"schemas/User\"} usage). I even added an explicit \"$id\": \"schemas/greeting\" to schemas/greeting.json to test id-based resolution. Every single variant produced 'unresolved $ref' from noda_validate_config ... This means the crud example in noda_get_examples itself demonstrates syntax that does not validate -- the gap is not a builder search failure, it's a real absence of working functionality/documentation reachable through MCP tools."

## Suggested fix
Either fix $ref resolution so the crud example's own syntax validates, or document the exact resolution rule (base path, extension, and whether it differs between `transform.validate`'s node-level schema and route-level `body.schema`) in a discoverable place — e.g. in `noda_get_config_schema(route)`'s `body.schema` description — and correct/annotate the crud example accordingly.

## Document the workflow-expression auth context object (auth.sub, auth.roles)

- **Labels:** documentation, component:engine
- **Hit by:** 04-auth  (impact: 1)

## Context
Workflows that gate behavior on the authenticated caller's identity or role (e.g. an admin-only endpoint) need to read claims like the subject id and roles from inside node config expressions (`control.if` conditions, `auth.get_user.user_id`, etc.).

## What is missing or confusing
No node doc, example, or schema explicitly documents an `auth` context object available in workflow expressions. The only clue anywhere in the MCP surface is a side-remark in the root config schema's `security.casbin.description` ("the subject is the authenticated user id (auth.sub)"), which is about casbin policy subjects, not about workflow expression context in general — and even that only covers `auth.sub`. `auth.roles` (needed for role checks) is not mentioned anywhere on the MCP surface.

## How to reproduce
Brief: `04-auth` (build phase). Search `noda_get_node_schema` for all `auth.*` nodes, `noda_get_examples(pattern=auth)`, `noda_list_functions`, and `noda_get_config_schema` for root/route/workflow.

Evaluator evidence: "The ONLY place 'auth.sub' appears anywhere in the MCP surface is the single aside in root config_schema's security.casbin.description ... Critically, 'roles' (the second half of the builder's stated goal, needed to check for the admin role) is not mentioned ANYWHERE in the MCP surface -- not in that sentence, not in any node schema, not in any example ... sub is technically reachable (buried in an unrelated field's description), but roles/full claim shape is not documented or exemplified anywhere."

## Suggested fix
Document the full shape of the `auth` context object (which claims/fields are guaranteed present, whether custom JWT claims round-trip into it) in a discoverable place — the JWT config section of the root schema, or a dedicated node/context schema — not only as an aside inside the unrelated casbin description. Add an example workflow that reads `{{ auth.sub }}` / `{{ auth.roles }}` for the current caller.

Related to #228 (role-claim RBAC via `casbin.enforce`), which covers the middleware-layer role-matching gap but not this workflow-expression-context gap.

## Document per-plugin service config requirements (e.g. auth plugin's required `database` field and migration bootstrap step)

- **Labels:** documentation, component:plugin, component:config
- **Hit by:** 04-auth  (impact: 1)

## Context
Services are declared in `noda.json` as `services.<name> = {plugin, config}`. The `auth` plugin's node family (`auth.get_user`, `auth.create_user`, etc.) declares `service_deps` on services tagged `auth` and `database`, but says nothing about what the `auth` service's own `config` object must contain, or that a schema-setup step is required before it can be used.

## What is missing or confusing
Three distinct gaps converge on the same root cause — the auth plugin's setup requirements are invisible to every `noda_*` tool:

1. **No plugin registry.** There is no `noda_list_plugins`-equivalent tool enumerating valid `services.plugin` identifiers or their config schemas. `noda_get_config_schema(root)` types `services.<name>.config` as a bare `{"type": "object"}` with no enum and no plugin-specific schema.
2. **Undocumented required config field.** The `auth` plugin's service config requires a `"database"` key pointing at the project's db service name (`{"plugin": "auth", "config": {"database": "db"}}`), or boot fails with `service "auth": create failed: auth: 'database' (service name) is required`. No `noda_*` tool states this.
3. **Undocumented migration/bootstrap requirement.** The auth plugin's real schema (`auth_users`, `auth_sessions`, `auth_tokens`) is only ever generated by the `noda auth init` CLI command. A project that wires `auth.get_user` directly (without running `noda auth init`) has no migration for these tables and can never create a real user — `noda_get_examples(pattern=auth)` doesn't even demonstrate the built-in `auth.*` node family, only a hand-rolled JWT pattern.

## How to reproduce
Brief: `04-auth` (build and e2e phases; 3 separate findings). Configure `services.auth = {"plugin": "auth", "config": {}}` per the node schemas' `service_deps`, then boot: `DATABASE_URL=... REDIS_URL=... JWT_SECRET=... ./bin/noda start --config <project dir>`.

Evaluator evidence (missing `database` field): "Boot fails immediately with: `service \"auth\": create failed: auth: 'database' (service name) is required` ... No tool ... or resource reachable from the MCP surface states that the auth plugin's service config requires \"database\"."

Evaluator evidence (missing migration path): "The auth plugin's real schema (auth_users, auth_sessions, auth_tokens) is only ever generated by the `noda auth init` CLI command ... there is no migrations/*auth* file present. This project therefore has no path to create even a single real user ... noda_get_examples(pattern=auth) is actively misleading here: it shows a hand-rolled JWT flow against a self-defined 'users' table ... never mentioning the real auth plugin or its CLI bootstrap at all."

## Suggested fix
Add a `noda_list_plugins` (or similar) MCP tool/resource enumerating available service plugin types with their expected config schema, the same way `noda_list_nodes` enumerates node types — including the `auth` plugin's required `database` field and its `noda auth init` migration-bootstrap requirement.

Related to #226 (general migration-file-format docs gap, filed against brief 03-data) and #231 (scaffolded service shape mismatch) — this issue is specifically about the `auth` plugin's undocumented required config field and bootstrap step, which neither of those covers.

## noda_validate_config does not catch invalid or incomplete plugin-specific service configs

- **Labels:** bug, type:test, component:config, component:server
- **Hit by:** 04-auth  (impact: 1)

## Context
`noda_validate_config` is the documented pre-flight check before `noda start`. Service configs are declared as `{plugin, config}` where `config`'s shape depends on the plugin.

## What is missing or confusing
`noda_validate_config` performs structural/cross-reference validation only — it does not validate a service's `config` against its plugin's actual required schema, and does not verify that `plugin` names a real, registered plugin type. Two independent reproductions:

- A project with `services.db.plugin = "postgres"` (not a real registered plugin family name) and `services.auth.plugin = "auth"` with empty `config` both passed `noda_validate_config` as `valid: true`.
- A project with `services.auth.config = {}` (missing the required `"database"` key) passed `noda_validate_config` as `{"valid": true, "file_count": 8, ...}` with no warnings, yet the identical project fails to boot with `service "auth": create failed: auth: 'database' (service name) is required`.

This gives agents a false green light: they have no tool-surfaced signal that their service config is wrong until they attempt to boot the server.

## How to reproduce
Brief: `04-auth` (build and e2e phases). Run `noda_validate_config` against a project with `services.auth = {"plugin": "auth", "config": {}}` (missing `database`) or `services.db = {"plugin": "postgres", ...}` (wrong plugin family name).

Evaluator evidence: "mcp__noda__noda_validate_config on the 04-auth project returns {\"valid\": true, \"file_count\": 8, ...} with no warnings, even though noda.json's services.auth = {\"plugin\":\"auth\",\"config\":{}} is missing the required \"database\" key ... services[*].config is typed as a bare {\"type\":\"object\"} with no plugin-specific keying (no oneOf/allOf on plugin name, no required 'database' for plugin=='auth'), so the schema tool itself cannot express this constraint, and by extension neither can whatever validator noda_validate_config wraps."

## Suggested fix
Extend `noda_validate_config` (and the underlying config validator) to check `services.<name>.config` against the concrete plugin's registered config schema, and to reject unrecognized `plugin` identifiers, rather than treating `config` as an opaque object.

This is the validation-side counterpart to the documentation gap tracked separately (plugin config requirements are undiscoverable) — fixing the docs alone would not stop `noda_validate_config` from giving a false green light on a broken config.

## Two incompatible auth patterns (built-in auth.* plugin vs hand-rolled JWT) can be mixed without warning, breaking on the actual user table

- **Labels:** documentation, bug, component:plugin
- **Hit by:** 04-auth  (impact: 1)

## Context
Noda offers two ways to build authentication: the built-in `auth.*` plugin node family (`auth.get_user`, `auth.create_user`, etc.) backed by a `services.auth` plugin, and a hand-rolled JWT pattern (db.findOne/db.create + `util.jwt_sign`/bcrypt against a self-managed table) — which is the only pattern `noda_get_examples(pattern=auth)` actually demonstrates.

## What is missing or confusing
The built-in `auth.get_user` node's Go implementation hardcodes `db.Table("auth_users")` — it only ever works against the schema created by `noda auth init`. Nothing in `noda_get_node_schema('auth.get_user')` or `noda_list_nodes(category='auth')` mentions this hardcoded table name. Meanwhile `noda_get_examples(pattern=auth)` shows only the hand-rolled pattern, which uses a `users` table via `util.jwt_sign` — and never demonstrates the built-in `auth.*` nodes at all, so there's no example warning that the two patterns are mutually exclusive schemas. A project that mixes them (one workflow uses `auth.get_user`, another does `db.count` on a hand-authored `"users"` table) is silently accepted by `noda_validate_config` and only fails at runtime — either as a 500 (no such table) or, once fixed, an empty/incorrect result (the wrong table).

## How to reproduce
Brief: `04-auth` (e2e phase), endpoint `GET /admin/stats`.

Evaluator evidence: "mcp__noda__noda_get_node_schema('auth.get_user') and noda_list_nodes(category='auth') describe inputs/outputs ... but never mention the underlying table name, so nothing signals it's hardcoded to 'auth_users' rather than configurable/matching whatever table the project uses. Worse, mcp__noda__noda_get_examples(pattern='auth') returns only the hand-rolled JWT pattern ... and never demonstrates the built-in auth.* node family ... there is no example showing these two mechanisms are mutually exclusive schemas ... An agent following the MCP-provided example pattern (which uses table 'users') while also picking up auth.get_user from noda_list_nodes ... has no tool-surfaced way to discover the incompatibility before it manifests as a 500 in production."

## Suggested fix
Add an example demonstrating the built-in `auth.*` node family end-to-end (including its `auth_users`/`auth_sessions`/`auth_tokens` schema and the `noda auth init` bootstrap), and explicitly document in `noda_get_node_schema('auth.get_user')` (and siblings) that these nodes require the `noda auth init`-generated schema and are not compatible with a hand-rolled `users` table.

## Fix bundled auth example's control.if edges to use the node's real output names ('then'/'else', not 'true'/'false')

- **Labels:** documentation, bug
- **Hit by:** 04-auth  (impact: 1)

## Context
`control.if`'s node schema declares `outputs: ['then', 'else', 'error']`. The login workflow returned by `noda_get_examples(pattern=auth)` wires `check_user`/`check_password` control.if nodes with edges using `output: "true"` / `output: "false"` instead.

## What is missing or confusing
Copying the bundled example verbatim produces dead edges: the `false`-branch edges never fire because `control.if` never emits an output literally named `"true"`/`"false"`. This is exactly the class of bug that would silently break an admin-role gate built by copying the example pattern.

## How to reproduce
Brief: `04-auth` (build phase). Compare `noda_get_node_schema('control.if')` (`outputs: ['then','else','error']`) against the edges in the login workflow from `noda_get_examples(pattern=auth)` (`{"from":"check_user","to":"verify","output":"true"}`).

Evaluator evidence: "noda_get_node_schema(control.if) declares outputs=['then','else','error']. noda_get_examples(pattern=auth) returns a login workflow whose edges use output values 'true'/'false' ... which do not match the node's declared 'then'/'else' outputs -- these edges would never fire ... the bundled 'auth' example is genuinely self-contradictory."

## Suggested fix
Fix the auth example in `noda_get_examples` to use `then`/`else`, matching `control.if`'s actual declared outputs.

## noda_validate_config does not flag edges whose 'output' value isn't a declared output of the referenced node

- **Labels:** bug, type:test, component:engine, component:config
- **Hit by:** 04-auth  (impact: 1)

## Context
Edges in a workflow reference a source node's `output` (e.g. `control.if`'s `then`/`else`/`error`). This is the mechanism that determines which branch fires.

## What is missing or confusing
`noda_validate_config` does not check that an edge's `output` field is one of the referenced node type's declared outputs. A minimal reproduction — a `control.if` node wired with `output: 'true'`/`'false'` edges, mirroring the bundled auth example's bug — validates as `{"valid": true}` with zero errors, even though those edges can never fire. This means a structurally-plausible but functionally-dead workflow gets a clean bill of health from the one tool an agent would use as a pre-flight check.

## How to reproduce
Brief: `04-auth` (build phase). Build a project with a `control.if` node whose edges use `output: "true"` / `output: "false"` (instead of the node's real `then`/`else` outputs) and run `noda_validate_config`.

Evaluator evidence: "I reproduced a minimal project with a control.if node wired with output:'true'/'false' edges (mirroring the bundled example) and ran noda_validate_config against it: result was {\"valid\": true} with zero errors -- the validator does not check that an edge's 'output' value is one of the referenced node's declared outputs, so copying the buggy bundled example verbatim passes validation silently."

## Suggested fix
Have `noda_validate_config` (and the underlying workflow validator) check each edge's `output` against the referenced node type's declared `outputs` list and report an error/warning when it doesn't match — this would have caught the bundled auth example's own bug (see companion issue on that example) and prevents silent dead-branch workflows in general.

## Document the connections/*.json directory convention; validate_config error is misleading when it's not followed

- **Labels:** documentation, bug, component:config
- **Hit by:** 05-hard  (impact: 1)

## Context
A WebSocket connections endpoint (e.g. a realtime board feature) is configured via a `connections` section. `noda_get_config_schema('root')` embeds a full inline `connections` object, which reads as if it belongs directly inside `noda.json`.

## What is missing or confusing
In practice, `connections` config is only discovered when placed in a `connections/` subdirectory (e.g. `connections/board.json`), mirroring the `routes/`/`workflows/` convention. Embedding it inline in `noda.json`, or in a top-level `connections.json` file, is silently ignored: `noda_list_project_files` omits `connections` from its categorized output entirely in that case, and `noda_validate_config` reports a generic `references non-existent connections endpoint "board"` error with no indication that the config wasn't even discovered — indistinguishable from a genuine typo. No tool (`noda_get_config_schema`, `noda_get_examples`, `noda_list_project_files`, `noda_read_project_file`'s path examples, `noda_scaffold_project`'s default output) states this directory requirement anywhere.

## How to reproduce
Brief: `05-hard` (build phase). Embed `connections.endpoints.board` inside `noda.json` (as the root schema implies is valid) and run `noda_validate_config`.

Evaluator evidence: "built a test project embedding 'connections.endpoints.board' inside noda.json ... noda_validate_config on it fails with the exact generic message the builder reported: 'references non-existent connections endpoint \"board\"' -- no hint that the endpoint config wasn't even discovered ... Checked every plausible disclosure point ... unlike the 'crud' pattern in the same tool which explicitly annotates 'migration_up/migration_down fields ... are SQL (migrations/<timestamp>_create_users.up.sql / .down.sql)' ... noda_scaffold_project's default scaffold does not include a connections/ directory or file at all, so it never demonstrates the convention."

## Suggested fix
Either accept a top-level `connections` key inside `noda.json` (as the root schema implies), or have `noda_get_config_schema`/`noda_scaffold_project`/`noda_list_project_files` explicitly state the required `connections/*.json` directory convention. At minimum, `noda_validate_config` should report "no connections file found / endpoint not defined anywhere" instead of the generic "references non-existent endpoint" message.

Related to #229 (realtime/connections conceptual doc — subscription model, channel-pattern relationship, lifecycle), which covers the content of a connections doc but not this file-placement/discovery convention.

## Scaffolded .env JWT_SECRET default fails noda's own 32-byte minimum, crashing on first boot

- **Labels:** bug, documentation, component:server
- **Hit by:** 05-hard  (impact: 1)

## Context
`noda_scaffold_project` writes `.env` and `.env.example` with `JWT_SECRET=change-me-in-production` (25 bytes), described by the tool's success message as "a ready-to-use .env ... written from docker-compose defaults." `noda_get_examples(pattern=auth)` then instructs wiring this same variable into `security.jwt.secret` via `{{ $env("JWT_SECRET") }}`.

## What is missing or confusing
Noda enforces a 32-byte minimum on JWT secrets at boot time, but the scaffolded placeholder is 7 bytes short. Boot fails with `middleware "auth.jwt": auth.jwt: secret is shorter than 32 bytes; use a stronger secret` — a crash that only surfaces after the operator tries to start the server, not from anything on the MCP surface. `noda_get_config_schema(config_type='root')` documents `security.jwt.secret` only as a bare `type: string` with no minimum-length note, and `noda_validate_config` reports `valid: true` for a project with the too-short scaffolded secret in place, giving a false pre-boot green light.

## How to reproduce
Brief: `05-hard` (e2e phase). Run `noda_scaffold_project` on a clean directory, then boot the resulting project without editing `.env`.

Evaluator evidence: "noda_scaffold_project (freshly run against a clean dir) writes .env and .env.example containing JWT_SECRET=change-me-in-production verbatim -- confirmed this is tool-generated, not something the builder typed ... noda_validate_config on the builder's project (with the scaffolded secret in place) reports valid:true -- it does not catch the too-short secret pre-boot. So across scaffold output, schema docs, and example docs, there is no warning that the shipped placeholder is 7 bytes under noda's own 32-byte minimum; the only place this surfaces is the boot-time crash."

## Suggested fix
Have the scaffold tooling generate a placeholder secret that already satisfies the 32-byte minimum (e.g. a random 32+ byte value, or a longer descriptive placeholder), or document the minimum length directly in `.env.example` next to `JWT_SECRET` and in the root config schema's `security.jwt.secret` description.

Related to #231 (scaffolded service shape mismatch) — a different aspect of the same underlying pattern of scaffold output not matching what the runtime actually requires.
