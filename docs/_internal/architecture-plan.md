# Noda ŌĆö Architecture Plan

**Version**: 0.4.0
**Language**: Go
**Status**: Complete

---

## 1. What Is Noda

Noda is a configuration-driven API runtime for Go. It lets developers build production-grade REST APIs, background workers, scheduled jobs, real-time connections, and long-running stateful services entirely through JSON configuration, a visual workflow editor, and Wasm modules. No application code is required for standard API patterns.

The framework provides a graph-based workflow engine, a plugin system for external service integration, built-in authentication and authorization, and automatic OpenAPI documentation ŌĆö all configured from JSON files generated by a visual editor.

---

## 2. Design Principles

**Configuration is the single source of truth.** JSON config files define every route, workflow, middleware chain, auth rule, and service connection. Code exists only to interpret and execute configuration.

**Fail at load time, not at runtime.** Every config file is validated at startup. Missing service references, unknown node types, invalid expressions, and broken edges are caught before the first request is served.

**Explicit over implicit.** Every node that uses an external service must name the instance. No defaults, no magic resolution, no guessing.

**Reuse existing packages.** Noda wraps battle-tested Go libraries for every core concern ŌĆö Fiber for HTTP, GORM for database, Casbin for authorization, Expr for expressions, Afero for storage. We do not build what others have already built well.

**Standardize everything.** JSON config files, error responses, node interfaces, workflow structures, and test definitions all follow strict, documented schemas. Consistency across the entire framework reduces cognitive load.

**Docker-first development.** Noda depends on system libraries (libvips) and external services (PostgreSQL, Redis). The official development setup is a Docker image with everything pre-installed and a `docker-compose.yml` that brings up the full stack with one command.

---

## 3. Runtime Architecture

Noda compiles to a **single binary**. That binary contains five runtime components, controlled by flags at startup:

- **HTTP Server** ŌĆö serves API endpoints, OpenAPI docs, WebSocket/SSE connections
- **Worker Runtime** ŌĆö consumes events from Redis Streams and executes workflows
- **Scheduler Runtime** ŌĆö fires workflows on cron schedules with distributed locking
- **Connection Manager** ŌĆö manages WebSocket and SSE connections across instances
- **Wasm Runtime** ŌĆö runs stateful Wasm modules (game servers, chat bots, trading systems, simulations, AI agents) via Extism with a tick-based execution model and single-dispatch host API

The first four share the same **Workflow Engine**. The Wasm runtime runs independently but can trigger workflows asynchronously and use Noda services through its host API.

```
HTTP Request  ŌöĆŌöĆŌöÉ
Event Message ŌöĆŌöĆŌöżŌöĆŌöĆŌåÆ Trigger Mapping ŌöĆŌöĆŌåÆ $.input ŌöĆŌöĆŌåÆ Workflow Engine
Cron Tick     ŌöĆŌöĆŌöż
WS Message    ŌöĆŌöĆŌöś

Wasm Module   ŌöĆŌöĆŌåÆ noda_call / noda_call_async ŌöĆŌöĆŌåÆ Services (Storage, Cache, WebSocket, Stream, PubSub, HTTP)
                                               ŌöĆŌöĆŌåÆ System Ops (trigger_workflow, log, timers)
              ŌåÉŌöĆŌöĆ tick / command / query ŌåÉŌöĆŌöĆ Noda (tick loop + Workflow Engine)
```

In development, `noda dev` runs all components together with hot reload. In production, the same binary can be deployed as API-only, worker-only, scheduler-only, wasm-only, or all-in-one using flags.

## 4. Configuration Format

All configuration is **JSON**. No YAML, no mixing formats. JSON files are the primary output of the visual editor, which is the intended authoring tool. Files are machine-generated, machine-read, and git-diffable.

### 4.1 File Organization

Config is organized into directories by concern:

- `noda.json` ŌĆö root application config (ports, global middleware, plugin/service declarations)
- `schemas/` ŌĆö shared, reusable type definitions
- `routes/` ŌĆö HTTP route definitions
- `workflows/` ŌĆö workflow graph definitions (nodes, edges, canvas positions)
- `workers/` ŌĆö event worker definitions
- `schedules/` ŌĆö cron schedule definitions
- `connections/` ŌĆö WebSocket and SSE endpoint definitions
- `tests/` ŌĆö workflow test definitions
- `migrations/` ŌĆö database migration SQL files (up/down pairs)

### 4.2 Environment Overlays

Noda supports per-environment configuration overlays. The base config is loaded first, then an environment-specific overlay is merged on top:

```
noda.json                  ŌåÉ base config
noda.development.json      ŌåÉ merged on top in dev
noda.staging.json          ŌåÉ merged on top in staging
noda.production.json       ŌåÉ merged on top in production
```

The environment is set via `--env` flag or `NODA_ENV` environment variable. Overlays can override any value ŌĆö database URLs, secrets, rate limits, feature flags. The base config defines the structure; overlays customize for the environment.

### 4.3 Shared Schema Definitions

Reusable type definitions live in the `schemas/` directory. Routes and workflows reference them to avoid repetition:

```json
{
  "User": {
    "type": "object",
    "properties": {
      "id": { "type": "string", "format": "uuid" },
      "name": { "type": "string", "minLength": 2 },
      "email": { "type": "string", "format": "email" },
      "role": { "type": "string", "enum": ["user", "manager", "admin"] },
      "created_at": { "type": "string", "format": "date-time" }
    },
    "required": ["id", "name", "email", "role"]
  },
  "Pagination": {
    "type": "object",
    "properties": {
      "page": { "type": "integer" },
      "limit": { "type": "integer" },
      "total": { "type": "integer" }
    }
  }
}
```

Routes reference these with `$ref`:

```json
{
  "response": {
    "200": {
      "schema": { "$ref": "schemas/User" }
    }
  }
}
```

### 4.4 Config Composition

To avoid repetition across routes, Noda supports **middleware presets** and **route groups**.

Middleware presets define reusable middleware chains:

```json
{
  "middleware_presets": {
    "authenticated": ["auth.jwt", "security.cors", "security.headers"],
    "admin_only": ["auth.jwt", "casbin.enforce", "security.cors", "security.headers"],
    "public": ["security.cors", "security.headers", "limiter.strict"]
  }
}
```

Route groups apply shared config to multiple routes:

```json
{
  "route_groups": {
    "/api/v1/admin": {
      "middleware_preset": "admin_only",
      "tags": ["admin"]
    },
    "/api/v1/public": {
      "middleware_preset": "public",
      "tags": ["public"]
    }
  }
}
```

Routes within a group inherit the group's middleware and tags. Per-route config can extend or override group defaults.

## 5. Standardized Structures

Every JSON config file follows a strict schema. The visual editor enforces these schemas at authoring time, and Noda validates them at startup.

### 5.1 Workflow Structure

Every workflow file follows this structure:

```json
{
  "id": "create-user",
  "name": "Create User",
  "description": "Creates a new user with duplicate checking",
  "version": "1.0.0",
  "nodes": {
    "node-id": {
      "type": "prefix.action",
      "services": {
        "slot-name": "instance-name"
      },
      "as": "optional-output-alias",
      "position": { "x": 0, "y": 0 },
      "config": {}
    }
  },
  "edges": [
    { "from": "node-a", "output": "success", "to": "node-b" },
    { "from": "node-a", "output": "error", "to": "node-c" }
  ]
}
```

Required fields: `id`, `nodes`, `edges`. Every workflow has an `id`, every node has a `type`, every edge has `from` and `to`. The `output` field on edges defaults to `"success"` if omitted.

### 5.2 Node Structure

Every node follows the same shape regardless of type:

```json
{
  "type": "prefix.action",
  "services": {
    "slot-name": "instance-name"
  },
  "as": "output-alias",
  "position": { "x": 0, "y": 0 },
  "config": {}
}
```

- `type` ŌĆö always required. Format is always `prefix.action`.
- `services` ŌĆö maps slot names to service instance names. Every node that needs services must use this field with explicit slot names. Omitted for core nodes with no service dependencies.
- `as` ŌĆö optional. Overrides the default output key (which is the node ID).
- `position` ŌĆö canvas coordinates for the visual editor.
- `config` ŌĆö node-specific configuration. Each node type defines its own config schema.

Single service:
```json
{ "type": "db.query", "services": { "database": "main-db" } }
```

Multiple services:
```json
{
  "type": "image.resize",
  "services": {
    "source": "raw-uploads",
    "target": "processed-images"
  }
}
```

There is no shorthand. Every service reference is a named slot. This keeps the config consistent ŌĆö the visual editor always renders the same structure, and the engine validates the same way regardless of how many services a node needs.

Each node type declares **service slots** ŌĆö named dependencies that each require a specific plugin prefix. The engine validates at startup that every slot is filled and that each referenced service instance comes from a plugin matching the required prefix.

### 5.2.1 Named Outputs

Every node type declares which **outputs** it provides. Outputs are the named connection ports that edges attach to. Most nodes have two outputs: `success` and `error`. Control flow nodes have additional outputs:

- **Most nodes**: `success`, `error`
- **`control.if`**: `then`, `else`, `error`
- **`control.switch`**: statically named branches from config (e.g., `"admin"`, `"user"`, `"guest"`), plus `default` and `error`. Case names must be string literals ŌĆö expressions are not allowed because outputs must be known at startup for graph validation and visual editor rendering.
- **`control.loop`**: `done` (fires when all iterations complete, carries collected results), `error`
- **`workflow.run`**: dynamic outputs collected from the sub-workflow's `workflow.output` nodes, plus `error` (see Section 8.5)

The visual editor reads each node's output declarations to render the correct connection ports. Edges reference outputs by name:

```json
{
  "edges": [
    { "from": "check-role", "output": "admin", "to": "admin-flow" },
    { "from": "check-role", "output": "user", "to": "user-flow" },
    { "from": "check-role", "output": "default", "to": "guest-flow" },
    { "from": "check-role", "output": "error", "to": "handle-error" }
  ]
}
```

If `output` is omitted on an edge, it defaults to `"success"`.

### 5.3 Route Structure

Every route follows this shape:

```json
{
  "id": "get-user",
  "method": "GET",
  "path": "/api/v1/users/:id",
  "summary": "Get user by ID",
  "tags": ["users"],
  "middleware": ["auth.jwt"],
  "params": { "schema": { "$ref": "schemas/UserParams" } },
  "query": { "schema": {} },
  "body": { "schema": {}, "content_type": "application/json" },
  "trigger": {
    "workflow": "get-user",
    "input": {}
  },
  "response": {
    "200": { "description": "Success", "schema": { "$ref": "schemas/User" } },
    "404": { "description": "Not found" }
  }
}
```

### 5.4 Worker Structure

```json
{
  "id": "process-notifications",
  "services": {
    "stream": "main-stream"
  },
  "subscribe": {
    "topic": "user.created",
    "group": "notification-workers"
  },
  "concurrency": 5,
  "middleware": ["worker.log", "worker.timeout"],
  "trigger": {
    "workflow": "send-welcome-email",
    "input": {
      "email": "{{ message.payload.email }}",
      "name": "{{ message.payload.name }}"
    }
  },
  "dead_letter": {
    "topic": "notifications.failed",
    "after": 3
  }
}
```

The `services.stream` field references which stream service instance the worker consumes from. This is required ŌĆö the engine validates at startup that the referenced stream service exists.

Note: workers also support **middleware** for cross-cutting concerns ŌĆö logging, error handling, metrics, timeout. Same concept as HTTP middleware, applied to message processing.

### 5.5 Schedule Structure

```json
{
  "id": "cleanup-tokens",
  "services": {
    "lock": "app-cache"
  },
  "cron": "0 */6 * * *",
  "timezone": "UTC",
  "description": "Remove expired refresh tokens",
  "trigger": {
    "workflow": "cleanup-tokens",
    "input": {}
  },
  "lock": {
    "enabled": true,
    "ttl": "300s"
  }
}
```

The `services.lock` field references a cache service instance used for distributed locking. When `lock.enabled` is true, the scheduler acquires a lock via the cache service (using atomic set-if-not-exists) before executing the job. This ensures only one instance executes each scheduled job. The `services.lock` field is required when `lock.enabled` is true.

## 6. Standardized Error Responses

Every error returned by Noda ŌĆö whether from middleware, validation, the workflow engine, or a node ŌĆö follows the same structure:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": [
      {
        "field": "email",
        "message": "must be a valid email address"
      }
    ],
    "trace_id": "abc-123-def"
  }
}
```

- `code` ŌĆö a machine-readable error code (e.g., `VALIDATION_ERROR`, `NOT_FOUND`, `UNAUTHORIZED`, `SERVICE_UNAVAILABLE`, `WORKFLOW_FAILED`)
- `message` ŌĆö a human-readable description
- `details` ŌĆö optional array of specific issues (field-level validation errors, node failure details)
- `trace_id` ŌĆö always present, links to the execution trace for debugging

Standard HTTP status codes are used. The `response.error` core node produces errors in this format. Middleware errors (auth, rate limit, CSRF) also conform.

This is the only error format Noda ever returns. No exceptions.

## 7. Trigger Mapping Layer

Every entry point normalizes its data into a unified `$.input` shape before the workflow executes. The trigger mapping is performed by the workflow engine as a pre-execution step ŌĆö it evaluates the input expressions against the raw trigger data (request, message, schedule metadata) and populates `$.input` before the first node runs.

An HTTP route maps selected request fields into `$.input`:

```json
{
  "path": "/api/users",
  "method": "POST",
  "trigger": {
    "workflow": "create-user",
    "input": {
      "email": "{{ body.email }}",
      "name": "{{ body.name }}"
    }
  }
}
```

An event worker maps message payload fields into the same `$.input`:

```json
{
  "subscribe": "user.requested",
  "trigger": {
    "workflow": "create-user",
    "input": {
      "email": "{{ message.payload.email }}",
      "name": "{{ message.payload.name }}"
    }
  }
}
```

The workflow engine evaluates these trigger expressions using the same Expr library and `{{ }}` syntax used in workflow nodes. The raw trigger data (`request`, `message`, `schedule`) is available only during this pre-execution step ŌĆö once `$.input` is populated, the raw trigger data is no longer accessible. This ensures workflows remain trigger-agnostic.

In addition to `$.input`, the execution context provides `$.auth` (populated by authentication middleware for HTTP triggers) and `$.trigger` (metadata about the trigger source ŌĆö type, timestamp, trace ID).

### 7.1 File Streams

When an HTTP route handles file uploads, the trigger config explicitly marks which input fields are file streams using a `files` array:

```json
{
  "path": "/api/users/:id/avatar",
  "method": "POST",
  "trigger": {
    "workflow": "upload-avatar",
    "input": {
      "user_id": "{{ params.id }}",
      "file": "{{ request.file }}"
    },
    "files": ["file"]
  }
}
```

Fields listed in `files` are passed through as raw file streams ŌĆö the trigger layer does not attempt to resolve, serialize, or buffer them. The `upload.handle` node knows to expect a stream on these fields. All other fields are resolved normally.

### 7.2 Raw Body Preservation

Many external webhooks (Stripe, GitHub, Shopify) require verifying a cryptographic signature against the raw, unparsed request body. By default, the trigger layer consumes the body to populate `$.input`, after which the raw bytes are no longer available.

When `raw_body` is set to `true` on the trigger config, the engine preserves the unparsed request body as `$.trigger.raw_body` (raw bytes) on the execution context:

```json
{
  "path": "/webhooks/stripe",
  "method": "POST",
  "trigger": {
    "workflow": "handle-stripe-webhook",
    "raw_body": true,
    "input": {
      "event": "{{ request.body }}"
    }
  }
}
```

The workflow can then use a `transform.validate` node or a dedicated verification step to check the HMAC signature against `{{ trigger.raw_body }}` before processing the event. Both `raw_body` and normal `input` mapping are available simultaneously ŌĆö the raw bytes are preserved alongside the parsed input.

### 7.3 HTTP Response Timing

For HTTP-triggered workflows, the response is sent to the client as soon as a `response.*` node fires ŌĆö not when the entire workflow completes. Any nodes that execute after the response node (e.g., `event.emit`, `util.log`) continue running asynchronously without holding the client connection open.

**Implementation:** Fiber pools its `*fiber.Ctx` ŌĆö once the HTTP handler returns, the context is recycled and unsafe to access. The engine uses a Go channel to signal the response. When a `response.*` node fires, it sends the `HTTPResponse` down the channel. The Fiber handler, which is waiting on this channel, writes the response to the client and returns. The workflow continues executing remaining nodes on the engine's goroutine pool. Background nodes only access the Noda execution context (which the engine owns), never the raw Fiber context. The raw trigger data (`request`) is already discarded after trigger mapping (Section 7), so background nodes have no path to the Fiber context.

If the workflow has **no `response.*` node**, the engine immediately returns `202 Accepted` and runs the workflow entirely in the background. This makes the route a fire-and-forget trigger ŌĆö useful for webhook receivers, async job submission, and any endpoint that doesn't need to return data.

If the workflow fails before reaching a `response.*` node, the engine returns the standardized error response with the appropriate HTTP status code.

## 8. Workflow Engine

The workflow engine is the core of Noda. It executes directed acyclic graphs of nodes, managing data flow, parallel execution, and error routing.

### 8.1 Graph Model

A workflow is defined as two separate structures: **nodes** and **edges**. This mirrors the N8N pattern and is optimized for visual editor read/write.

Nodes define what to do:

```json
{
  "nodes": {
    "fetch-user": {
      "type": "db.query",
      "services": { "database": "main-db" },
      "position": { "x": 100, "y": 200 },
      "config": {
        "query": "SELECT * FROM users WHERE id = $1",
        "params": ["{{ input.user_id }}"]
      }
    },
    "fetch-orders": {
      "type": "db.query",
      "services": { "database": "main-db" },
      "position": { "x": 100, "y": 400 },
      "config": {
        "query": "SELECT * FROM orders WHERE user_id = $1",
        "params": ["{{ input.user_id }}"]
      }
    },
    "build-response": {
      "type": "response.json",
      "position": { "x": 400, "y": 300 },
      "config": {
        "status": 200,
        "body": {
          "user": "{{ fetch-user }}",
          "orders": "{{ fetch-orders }}"
        }
      }
    }
  }
}
```

Edges define data flow:

```json
{
  "edges": [
    { "from": "fetch-user", "to": "build-response" },
    { "from": "fetch-orders", "to": "build-response" }
  ]
}
```

The `output` field is omitted here, so both default to `"success"` ŌĆö the data flows when the node succeeds.

### 8.2 Execution Rules

**Node outputs** are stored on the execution context under the node's ID. `fetch-user` produces a result accessible as `{{ fetch-user }}`. A node can override this with an `as` field ŌĆö `"as": "user"` makes the result available as `{{ user }}` instead.

**Parallelism is implicit.** The engine analyzes the graph at startup and determines which nodes can run concurrently. In the example above, `fetch-user` and `fetch-orders` have no dependency on each other, so they run in parallel. `build-response` depends on both, so it waits.

**Entry nodes** are nodes with no inbound edges. The engine starts execution at all entry nodes simultaneously.

**A workflow completes** when all terminal nodes (nodes with no outbound edges on their `success` output) have finished.

**Flow convergence is automatic.** When a node has multiple inbound edges, the engine determines which edges will actually fire based on the graph structure:

- **Parallel branches** (e.g., two entry nodes both leading to a downstream node): the engine waits for ALL inbound edges. This is an AND-join.
- **Conditional branches** (e.g., both the `then` and `else` outputs of a `control.if` eventually lead to the same node): the engine knows only one branch will fire and waits for whichever was activated. This is an OR-join.

The engine resolves this at graph compilation time by tracing each inbound edge back to its origin. Edges that share a common conditional ancestor are mutually exclusive ŌĆö the engine only waits for the branch that was taken. No explicit merge or join node is needed for flow convergence.

**Execution context memory management.** Node outputs stored on the execution context can be large (e.g., a `db.query` returning 50MB of rows). Since all expressions are parsed at startup, the engine knows exactly which downstream nodes reference each output. Once the last dependent node has executed, the engine evicts that output from the context map, allowing Go's garbage collector to reclaim the memory. This prevents memory bloat in high-throughput workflows with large intermediate data.

### 8.3 Error Handling

Every node has at least a `success` and `error` output. Edges reference which output they connect to:

```json
{
  "edges": [
    { "from": "fetch-user", "to": "build-response" },
    { "from": "fetch-user", "output": "error", "to": "handle-db-error" }
  ]
}
```

When a node succeeds, data flows along edges connected to its `success` output (the default when `output` is omitted). When it fails, data flows along edges connected to its `error` output. The error information (message, code, originating node) is available on the context for the error-handling node to use.

**If a node fails and has no edge on its `error` output, the entire workflow fails.** No silent error swallowing. The standardized error response is returned automatically.

### 8.4 Retry on Edges

Retries are configured on **edges**, not as separate nodes. An error edge can specify a retry policy ŌĆö the engine retries the source node before following the edge. If all retries are exhausted, the edge is followed.

```json
{
  "edges": [
    { "from": "fetch-data", "output": "error", "to": "handle-failure",
      "retry": { "attempts": 3, "backoff": "exponential", "delay": "1s" } }
  ]
}
```

If `fetch-data` fails, the engine retries it up to 3 times with exponential backoff. Only after all attempts are exhausted does data flow to `handle-failure`. If `fetch-data` succeeds on a retry, its success output fires normally.

**Retries respect the `context.Context` deadline.** If the context expires during a retry backoff sleep, the engine cancels the sleep immediately and follows the error edge ŌĆö no further attempts are made. This ensures that total workflow execution time is always bounded by the context deadline, regardless of retry configuration. For HTTP-triggered workflows, this prevents retries from exceeding API gateway or client timeouts.

The visual editor renders retries as a badge on the error edge ŌĆö no extra node cluttering the graph, no graph cycles. Edges without a `retry` field follow immediately on error. The `retry` field is only valid on edges where `"output": "error"` ŌĆö the engine rejects retry config on any other edge at startup.

### 8.5 Sub-Workflows

Workflows can invoke other workflows using the `workflow.run` core node. This enables reuse ŌĆö a "send notification" workflow used by 10 different parent workflows ŌĆö and composition of complex flows from smaller, testable units.

```json
{
  "type": "workflow.run",
  "config": {
    "workflow": "check-inventory",
    "input": {
      "product_id": "{{ validated-order.product_id }}",
      "quantity": "{{ validated-order.quantity }}"
    }
  }
}
```

The sub-workflow receives its own `$.input` from the parent's config.

#### Sub-Workflow Outputs

Sub-workflows define their outputs using `workflow.output` nodes ŌĆö terminal nodes that declare a named output and the data to return:

```json
{
  "type": "workflow.output",
  "config": {
    "name": "in-stock",
    "data": { "quantity": "{{ check-stock.remaining }}" }
  }
}
```

A sub-workflow can have multiple `workflow.output` nodes across different branches. For example, a "check-inventory" sub-workflow might have:

- `workflow.output` with `name: "in-stock"` ŌĆö the product is available
- `workflow.output` with `name: "out-of-stock"` ŌĆö the product is unavailable
- `workflow.output` with `name: "error"` ŌĆö something went wrong

Only one `workflow.output` node fires per execution. Whichever fires determines which output the `workflow.run` node emits in the parent workflow. This is the same contract as control flow nodes ŌĆö `control.if` fires exactly one of `then`/`else`, `control.switch` fires exactly one case, and `workflow.run` fires exactly one of its dynamic outputs.

The engine enforces this at startup by validating that all `workflow.output` nodes within a sub-workflow are on **mutually exclusive branches** ŌĆö they must all descend from conditional splits (`control.if`, `control.switch`), never from parallel branches where multiple outputs could fire simultaneously. If two `workflow.output` nodes could both be reached in a single execution, the sub-workflow fails validation and the application does not start.

The parent wires edges to each possible output:

```json
{
  "edges": [
    { "from": "check-inventory", "output": "in-stock", "to": "reserve-item" },
    { "from": "check-inventory", "output": "out-of-stock", "to": "notify-restock" },
    { "from": "check-inventory", "output": "error", "to": "handle-error" }
  ]
}
```

The `workflow.run` node has **dynamic outputs** ŌĆö its factory reads the referenced sub-workflow, collects all `workflow.output` node names, and returns those as its `Outputs()`. The engine validates at startup that every edge references a valid output name, and every `workflow.output` name within a sub-workflow is unique.

If a sub-workflow node fails and has no error edge, the engine fires the `workflow.run` node's `"error"` output in the parent (this output is always present, even if the sub-workflow doesn't define a `workflow.output` named "error").

### 8.6 Transactions

Database transactions that span multiple operations use the `workflow.run` node with `transaction: true`. This wraps an entire sub-workflow in a database transaction.

```json
{
  "type": "workflow.run",
  "services": {
    "database": "main-db"
  },
  "config": {
    "workflow": "create-order-steps",
    "input": {
      "order": "{{ validated-order }}"
    },
    "transaction": true
  }
}
```

When `transaction: true`, the engine creates a database transaction from the referenced service. All `db.*` nodes inside the sub-workflow receive the transaction-scoped connection instead of the normal connection pool. If the sub-workflow completes successfully, the transaction commits. If any node fails, the transaction rolls back and the `workflow.run` node emits an error.

The sub-workflow itself doesn't know it's running inside a transaction. Its nodes declare their `db` service slot as normal. The engine transparently swaps the connection underneath at the sub-workflow boundary.

This design keeps transactions explicit and visible in the graph ŌĆö the parent workflow shows a `workflow.run` node with `transaction: true`, and the visual editor can render it distinctly.

### 8.7 Loops

The `control.loop` node iterates over a collection by invoking a sub-workflow once per item. Each iteration runs the sub-workflow sequentially ŌĆö the current iteration must complete before the next one starts.

```json
{
  "type": "control.loop",
  "config": {
    "collection": "{{ fetch-users }}",
    "workflow": "process-single-user",
    "input": {
      "user": "{{ $item }}",
      "index": "{{ $index }}"
    }
  }
}
```

`$item` is the current element from the collection. `$index` is the zero-based iteration index. These are special variables available only within the loop's input mapping.

The sub-workflow runs with its own `$.input` populated from the loop's `input` config. It uses `workflow.output` nodes to define its result, just like any other sub-workflow. The loop collects the output data from each iteration into an array.

When all iterations complete, the `done` output fires with the collected results array. If any iteration fails (the sub-workflow errors with no error edge), the loop immediately fires its `error` output ŌĆö remaining iterations are skipped. To continue on failure, the sub-workflow should handle its own errors internally and always reach a `workflow.output` node.

### 8.8 Execution Tracing

The workflow engine records execution traces at two levels depending on the runtime mode.

**Development mode ŌĆö full live streaming:**

In dev mode, the engine captures a complete trace for every workflow execution and streams it in real time over a WebSocket endpoint. The visual editor connects to this endpoint and visualizes the execution live ŌĆö highlighting nodes as they execute, showing data flowing along edges, and marking failures.

Each trace event includes:

- **Node entered** ŌĆö which node started executing, the resolved input data it received
- **Node completed** ŌĆö which node finished, the output data it produced, execution duration
- **Node failed** ŌĆö which node errored, the error details (message, code, stack)
- **Edge followed** ŌĆö which edge was taken, from which node's output to which target node
- **Retry attempted** ŌĆö which edge triggered a retry, attempt number, delay before next attempt
- **Workflow started** ŌĆö trigger source, input data, trace ID
- **Workflow completed** ŌĆö final status (success/error), total duration

This gives the developer a complete picture: they fire a request, switch to the editor, and watch the workflow execute node by node with full data visibility at every step.

Full tracing is **only available in dev mode**. It is never active in production ŌĆö the data capture overhead and the risk of exposing sensitive data in traces make it unsuitable.

**Production mode ŌĆö lightweight tracing:**

In production, the engine emits lightweight OpenTelemetry spans for each workflow execution. These capture:

- Which nodes executed and in what order
- Execution duration per node
- Success or failure status per node
- Error messages (without full input/output data)
- The trace ID for correlation across services

These traces feed into standard observability tools (Grafana, Jaeger, Datadog) for production debugging and performance monitoring. No full data payloads, no streaming ŌĆö just structured telemetry.

## 9. Expression Engine

Noda uses the **Expr** library (`expr-lang/expr`) for all expression evaluation. No custom expression syntax. What Expr supports, Noda supports ŌĆö directly and without modification.

`{{ }}` serves only as a delimiter to mark "this string contains an expression." Everything inside is native Expr syntax, evaluated by the Expr library against the current execution context.

Examples:

- `{{ input.name }}` ŌĆö simple path access
- `{{ input.page * input.limit }}` ŌĆö arithmetic
- `{{ input.role == "admin" ? "full" : "limited" }}` ŌĆö ternary
- `"Hello {{ input.name }}, you have {{ len(orders) }} orders"` ŌĆö string interpolation

Expressions are **compiled once at startup** (when workflows are loaded) and **evaluated per execution** against the context. This gives fast per-request performance with no parsing overhead.

The execution context exposed to Expr includes:

- `input` ŌĆö the normalized trigger input
- `auth` ŌĆö authentication data (JWT claims, user ID, roles)
- `trigger` ŌĆö metadata about the trigger source
- All node outputs by their ID (or `as` name)

Custom functions can be registered by plugins to extend the expression vocabulary.

## 10. Authentication

Authentication is handled as **middleware** on HTTP routes, not as a workflow concern. It runs before the workflow is triggered and populates `$.auth` on the execution context.

Noda uses Fiber's official JWT middleware package (`gofiber/contrib/jwt`), built on `golang-jwt/jwt/v5`. The middleware validates tokens, extracts claims, and makes them available to the workflow.

Configuration is in the root config:

```json
{
  "security": {
    "jwt": {
      "secret": "{{ $env('JWT_SECRET') }}",
      "algorithm": "HS256",
      "token_lookup": "header:Authorization"
    }
  }
}
```

Routes reference it as middleware:

```json
{
  "middleware": ["auth.jwt"]
}
```

API key authentication is also available via Fiber's built-in KeyAuth middleware, configured the same way.

## 11. Authorization

Authorization is handled by **Casbin**, a production-grade access control library that supports ACL, RBAC, ABAC, multi-tenant RBAC, and more ŌĆö all from configuration.

Casbin evaluates authorization in the `{subject, object, action}` form: "Can this user access this path with this method?" The access control model and policies are defined in config, matching Noda's philosophy.

Casbin already has an official Fiber middleware package. Noda wraps it with a config layer so users define their authorization model and policies in JSON, and the framework wires it into Fiber at startup.

Authorization runs after authentication ŌĆö the JWT middleware populates the user identity, then Casbin checks whether that identity has permission for the requested resource.

## 12. Middlewares

Noda does not implement custom middleware. All middleware comes from Fiber's built-in collection and official third-party packages, configured from JSON.

Available middlewares:

**Security**: CORS, CSRF, Helmet (security headers), Rate Limiter, Encrypt Cookie, KeyAuth (API keys), JWT Auth, Casbin Auth

**Operational**: Recover (panic recovery), Logger, RequestID, Healthcheck (liveness/readiness probes), Timeout, Idempotency

**Performance**: Cache (HTTP response caching), Compression (gzip/brotli/zstd), ETag

Middlewares are applied at three levels:

- **Global** ŌĆö defined in the root config, applied to all routes
- **Per-group** ŌĆö defined on route groups via middleware presets, applied to all routes in the group
- **Per-route** ŌĆö defined on individual route configs, can extend or override group defaults

Workers also support middleware-like hooks for cross-cutting concerns: logging, timeout, error handling, and metrics. These follow the same configuration pattern.

## 13. Plugin & Service Architecture

Plugins are the bridge between the workflow engine and external systems. They follow a two-layer model.

### 13.1 Plugin (Type Layer)

A plugin is a **type** ŌĆö it knows how to connect to a specific kind of external system and provides workflow nodes for interacting with it. A plugin is registered once and owns a **prefix** for its nodes.

Examples:

- The `postgres` plugin owns the `db` prefix ŌåÆ provides `db.query`, `db.exec`, `db.create`, `db.update`, `db.delete`
- The `cache` plugin owns the `cache` prefix ŌåÆ provides `cache.get`, `cache.set`, `cache.del`
- The `stream` plugin owns the `stream` prefix ŌåÆ provides stream infrastructure for workers
- The `pubsub` plugin owns the `pubsub` prefix ŌåÆ provides real-time fan-out and cross-instance sync
- The `storage` plugin owns the `storage` prefix ŌåÆ provides `storage.read`, `storage.write`, `storage.delete`, `storage.list`

### 13.2 Service Instance (Connection Layer)

A service instance is a **named connection** created from a plugin with specific configuration. Multiple instances per plugin are allowed:

```json
{
  "services": {
    "main-db": {
      "plugin": "postgres",
      "config": { "url": "postgres://localhost/main" }
    },
    "analytics-db": {
      "plugin": "postgres",
      "config": { "url": "postgres://localhost/analytics" }
    },
    "app-cache": {
      "plugin": "cache",
      "config": { "url": "redis://localhost:6379/0" }
    },
    "main-stream": {
      "plugin": "stream",
      "config": { "url": "redis://localhost:6379/0" }
    },
    "realtime": {
      "plugin": "pubsub",
      "config": { "url": "redis://localhost:6379/0" }
    }
  }
}
```

### 13.3 Explicit Service References

Every workflow node that uses a service **must** explicitly reference which service instance fills each slot. No defaults, no shorthand, no fallback.

Each node type declares **service slots** ŌĆö named dependencies, each requiring a service from a specific plugin prefix. The workflow config maps slot names to actual service instance names using the `services` field:

```json
{
  "type": "db.query",
  "services": { "database": "analytics-db" }
}
```

```json
{
  "type": "image.resize",
  "services": {
    "source": "raw-uploads",
    "target": "processed-images"
  }
}
```

The engine validates at startup that every declared slot is filled and that each referenced service instance comes from a plugin matching the slot's required prefix.

### 13.4 Service Registry

The service registry is the central lookup for all initialized service instances. It is **not flat** ŌĆö it understands the relationship between plugins, prefixes, and instances.

The registry knows:

- Which plugin owns which prefix
- Which service instances exist for each plugin
- The health status of each instance

Any part of the system ŌĆö workflow engine, worker runtime, scheduler, connection manager ŌĆö retrieves services from the registry by name.

### 13.5 Internal Service Registration

Not all services are declared in the `services` config block. Internal services are auto-registered by the engine from their respective config sections:

- **Connection endpoints** ŌĆö each endpoint key in `connections.endpoints` (e.g., `"user-updates"`, `"live-feed"`) registers as a service instance under the `ws` or `sse` prefix based on its `type`. Workflow nodes reference them by endpoint name: `"services": { "connections": "user-updates" }`.
- **Wasm runtimes** ŌĆö each key in `wasm_runtimes` (e.g., `"game"`, `"discord-bot"`) registers as a service instance under the `wasm` prefix. Workflow nodes reference them by runtime name: `"services": { "runtime": "game" }`.

Internal services participate in the same registry as external services. The engine validates references to them the same way ŌĆö if a node references `"services": { "connections": "user-updates" }` but no endpoint named `"user-updates"` is configured, that's a startup error.

External services (database, cache, storage, stream, pubsub) are always declared in the `services` config block. Internal services are never declared there ŌĆö they are derived from their own config sections.

### 13.6 Startup Validation

At startup, the engine:

1. Loads all plugins and collects their prefixes
2. Initializes all service instances from config
3. Scans all workflows and collects every node type referenced
4. Verifies that every prefix has a loaded plugin
5. Verifies that every service reference has a configured instance
6. Reports all errors with clear messages before starting

If validation fails, the application does not start.

### 13.7 Dual-Plugin Prefixes

Some capabilities use two plugins that share a prefix:

- **`storage`**: The `storage` plugin (`plugins/storage/`) manages Afero service instances (connections to local/S3 filesystems). The `core.storage` plugin (`plugins/core/storage/`) provides workflow nodes (`storage.read`, `storage.write`, `storage.delete`, `storage.list`). This works because the service plugin has `HasServices=true, Nodes=nil` and the core plugin has `HasServices=false, Nodes=[4]` ŌĆö the registry distinguishes them by role.

## 14. Database

Noda uses **GORM** (`gorm.io/gorm`) as its ORM layer, wrapped by the `postgres` (or `mysql`, `sqlite`) plugin.

GORM provides:

- Support for PostgreSQL, MySQL, SQLite, and SQL Server
- Automatic migrations
- Relationships, transactions, and hooks
- Query building with `map[string]any` (no Go struct definitions required)

Because Noda is config-driven, GORM's ability to work with dynamic maps instead of predefined structs is essential. Users define their data models in JSON config, and the plugin translates operations into GORM calls.

The database plugin provides nodes under the `db` prefix: `db.query`, `db.exec`, `db.create`, `db.update`, `db.delete`.

Transactions spanning multiple nodes are handled by the `workflow.run` node with `transaction: true` (see Section 8.6).

For migrations, GORM's auto-migrate can be used for rapid prototyping in development. For production, Noda provides explicit migration management via the `noda migrate` CLI:

- `noda migrate create [name]` ŌĆö generates a timestamped pair of SQL files (`YYYYMMDDHHMMSS_name.up.sql` and `YYYYMMDDHHMMSS_name.down.sql`) in a `migrations/` directory
- `noda migrate up` ŌĆö applies all pending migrations in order
- `noda migrate down` ŌĆö rolls back the last applied migration
- `noda migrate status` ŌĆö shows which migrations have been applied

Migration files are raw SQL written by the developer. Because Noda models data in JSON config rather than Go structs, automatic schema diff generation is not possible ŌĆö the developer is responsible for authoring the SQL. The visual editor could eventually generate migration SQL from schema changes, but migrations are always reviewed and committed as SQL files.

## 15. Events

Noda supports two event delivery mechanisms. Each is its own plugin with its own prefix, both backed by Redis:

### 15.1 Stream Plugin (`stream` prefix) ŌĆö Durable Job Processing

Streams are used by the **worker runtime** for guaranteed, durable message processing. They provide consumer groups, message acknowledgment, replay capability, and dead letter handling. The stream plugin wraps Redis Streams.

Use case: "Process this job reliably. If a worker crashes, another picks it up."

### 15.2 PubSub Plugin (`pubsub` prefix) ŌĆö Real-Time Fan-Out

PubSub is used for real-time broadcasting where persistence is not needed. All subscribers receive the message simultaneously. No acknowledgment, no replay. The pubsub plugin wraps Redis PubSub.

Use case: "Something just happened. Tell everyone who's listening right now."

### 15.3 Cross-Instance Sync

PubSub serves a critical infrastructure role: **synchronizing the connection manager across multiple Noda instances**. When a workflow on instance A sends a WebSocket message to channel `user.123`, but that user is connected to instance B, the connection manager looks up the channel in a Redis routing table to find instance B and publishes the message to that specific instance via PubSub. Instance B receives it and delivers to the local connection. See Section 19.5 for details on the routing table.

### 15.4 Unified Event Node

A single `event.emit` node handles both mechanisms. It declares optional service slots for both `stream` and `pubsub`. The delivery mode is specified in the node's config, and only the matching slot is required:

```json
{
  "type": "event.emit",
  "services": {
    "stream": "main-stream"
  },
  "config": {
    "topic": "user.created",
    "mode": "stream",
    "payload": "{{ created-user }}"
  }
}
```

The `mode` field determines whether the event is published to a Stream (durable, for workers) or PubSub (real-time, for broadcast/sync). Mode must be a static string (`"stream"` or `"pubsub"`), not an expression ŌĆö the engine needs to know at startup which service slot is required. The engine validates at startup that the slot matching the configured mode is present.

### 15.5 Service Configuration

Stream and PubSub are configured as separate service instances:

```json
{
  "services": {
    "main-stream": {
      "plugin": "stream",
      "config": { "url": "redis://localhost:6379/0" }
    },
    "realtime": {
      "plugin": "pubsub",
      "config": { "url": "redis://localhost:6379/0" }
    }
  }
}
```

They can share the same Redis instance (same URL) or use different ones. The separation is logical ŌĆö different prefixes, different plugins, independent configuration.

## 19. WebSockets & Server-Sent Events

Real-time connections are a **core internal service**, not a plugin. The connection manager tracks all open WebSocket and SSE connections and handles cross-instance synchronization via Redis PubSub.

### 19.1 Configuration

WebSocket and SSE endpoints are defined in JSON config. The connection manager requires a `pubsub` service for cross-instance synchronization, and each endpoint registers as a service instance that nodes can reference.

```json
{
  "connections": {
    "sync": {
      "pubsub": "realtime"
    },
    "endpoints": {
      "user-updates": {
        "type": "websocket",
        "path": "/ws/updates",
        "middleware": ["auth.jwt"],
        "channels": {
          "pattern": "user.{{ auth.sub }}",
          "max_per_channel": 5
        },
        "ping_interval": "30s",
        "max_message_size": "64kb",
        "on_connect": "ws-on-connect",
        "on_message": "ws-on-message",
        "on_disconnect": "ws-on-disconnect"
      },
      "live-feed": {
        "type": "sse",
        "path": "/events/feed",
        "middleware": ["auth.jwt"],
        "channels": {
          "pattern": "feed.{{ input.topic }}"
        },
        "retry": "5s",
        "heartbeat": "15s",
        "on_connect": "sse-on-connect"
      }
    }
  }
}
```

Each endpoint key (e.g., `"user-updates"`, `"live-feed"`) automatically registers as a service instance under the `ws` or `sse` prefix, based on the endpoint's `type`. Workflow nodes reference specific endpoints by name:

```json
{
  "type": "ws.send",
  "services": { "connections": "user-updates" },
  "config": {
    "channel": "user.{{ fetch-user.id }}",
    "data": "{{ build-payload }}"
  }
}
```

The `sync.pubsub` field references a pubsub service instance for cross-instance message delivery. At startup, the engine validates that the referenced service exists. If any connection endpoints are defined but `sync.pubsub` is missing, the application fails to start.

For single-instance deployments, the pubsub service is still required ŌĆö this keeps the config consistent across environments and avoids deployment errors when scaling.

### 19.2 Lifecycle Events as Workflows

Connection lifecycle events ŌĆö `on_connect`, `on_message`, `on_disconnect` ŌĆö reference workflows by ID. They are entry points into the workflow engine, just like HTTP routes or event subscriptions.

### 19.3 Outbound Delivery via Nodes

Workflow nodes `ws.send` and `sse.send` push data to connected clients by targeting a channel name on a specific endpoint (see example above in Section 19.1).

### 19.4 Wildcard Channels

Channels support wildcard patterns for broadcasting:

- `user.123` ŌĆö send to a specific user
- `user.*` ŌĆö broadcast to all user channels
- `feed.*` ŌĆö broadcast to all feed subscribers
- `*` ŌĆö broadcast to all connected clients

### 19.5 Multi-Instance Scaling

When running multiple Noda instances, the connection manager tracks which instance holds which connections using a **routing table in Redis**. When a client connects, the instance registers the connection's channel in a Redis hash mapping `channel ŌåÆ instance_id`. When a client disconnects, the entry is removed.

When a workflow node sends a message to a channel (e.g., `ws.send` to `user.123`), the connection manager looks up the routing table to determine which instance holds that connection and publishes the message only to that instance via Redis PubSub. This is **targeted delivery** ŌĆö not broadcast.

For wildcard channels (`user.*`, `*`), the connection manager queries the routing table for all matching channels, groups them by instance, and publishes once per instance. This is still more efficient than broadcasting every message to every instance.

The routing table entries have a TTL slightly longer than the ping interval. If an instance crashes without cleaning up, entries expire automatically and the routing table self-heals.

**Implementation note:** Redis HASH does not support pattern-based key lookups natively. Wildcard channel resolution (`user.*`, `*`) will require either SCAN-based iteration, a secondary sorted set index for pattern matching, or a hierarchical key structure that enables prefix queries. The specific approach should be chosen during implementation based on expected connection counts and wildcard usage patterns.

## 21. Wasm Runtime

The Wasm runtime is Noda's fifth runtime component. It is a **general-purpose host for stateful Wasm modules** ŌĆö game servers, chat bots, trading systems, IoT gateways, simulations, AI agents, or custom protocol bridges.

Unlike the other runtimes, the Wasm runtime does not execute workflows. It runs user-provided Wasm binaries that contain their own logic. Noda provides the infrastructure layer; the Wasm module provides the application logic.

The Wasm runtime is built on **Extism**, which handles memory management, data passing, and host function linking. Module authors use the Extism PDK for their language. The full contract is defined in the **Wasm Host API** document.

### 21.1 Architecture

Noda uses a **tick-based execution model**. Instead of the module running its own infinite loop, Noda calls the module's exported functions at a configured tick rate. Between ticks, the module's Wasm instance stays alive ŌĆö all linear memory, heap allocations, global variables, and static data persist across calls.

**Responsibilities of the Wasm module:**
- All application logic (game mechanics, bot commands, trading strategies, protocol handling)
- Internal state management (game world, session data, entity systems) ŌĆö stored in Wasm linear memory
- Processing events delivered by Noda each tick
- Calling Noda services via the `noda_call` host function

**Responsibilities of Noda:**
- Calling the module at the configured tick rate
- Accumulating events (WebSocket messages, workflow commands) and delivering them in the tick input
- Managing outbound WebSocket connections (connect, reconnect, heartbeat, buffer incoming messages)
- Providing the host API for storage, cache, events, logging, and workflow triggering
- Authentication and authorization for all client connections
- Network security ŌĆö controlling which external hosts the module can reach

### 21.2 Module Exports

The module exports functions that Noda calls:

| Export | Called by | Description |
|---|---|---|
| `initialize` | Noda at startup | Receives config + service manifest. Allocate state, load persisted data. |
| `tick` | Noda at tick_rate Hz | Receives delta time, client messages (with user identity), outbound WebSocket messages, connection lifecycle events (connect/disconnect/reconnect), workflow commands, async call responses, and fired timers. |
| `command` | `wasm.send` workflow node | Receives data from workflow (immediate delivery, between ticks). Optional ŌĆö if not exported, commands are delivered in the next tick. |
| `query` | `wasm.query` workflow node | Receives query data, returns response synchronously. Must be fast. |
| `shutdown` | Noda at graceful stop | Persist state, release resources. |

### 21.3 Host API ŌĆö `noda_call` and `noda_call_async`

The module communicates with Noda through two host functions:

**Synchronous:** `noda_call(service, operation, payload) ŌåÆ response` ŌĆö blocks until complete. For fast operations: cache reads, WebSocket sends, logging, event emission.

**Asynchronous:** `noda_call_async(service, operation, payload, label)` ŌĆö returns immediately. Noda performs the operation on its own goroutine. The result is delivered in the next tick's `responses` field, keyed by the label. For slow operations: outbound HTTP requests, large storage reads/writes.

The first argument is the service instance name (or `""` for system operations). The module can only call services listed in its manifest. Calls to unauthorized services return `PERMISSION_DENIED`. Database access is not available ŌĆö modules that need database operations trigger workflows containing database nodes.

### 21.4 Encoding

The wire format for all data crossing the Wasm boundary is configurable per module: JSON (`"json"`, default) for development and low-frequency modules, or MessagePack (`"msgpack"`) for high-frequency modules (30+ Hz) where serialization overhead matters. The Noda PDK handles encoding transparently ŌĆö switching is a config change, not a code change.

### 21.5 Outbound Connections

For modules that need persistent outbound connections (Discord gateway, exchange WebSocket feeds), Noda manages the connection lifecycle. The module requests a connection via `noda_call("", "ws_connect", ...)` during initialization.

Noda handles:
- TCP/TLS establishment and reconnection on failure
- WebSocket ping/pong keepalive
- Protocol-level heartbeats (configurable per connection)
- Buffering incoming messages for delivery in the next tick

The module receives clean application-level messages in the tick's `incoming_ws` array. It never sees raw WebSocket frames or protocol details. Outbound connections are only allowed to hosts in the module's `allow_outbound.ws` whitelist.

### 21.6 Interaction with Workflows

Workflows interact with running Wasm modules through two nodes:

**`wasm.send`** ŌĆö async, fire and forget. The workflow does not wait for a response:

```json
{
  "type": "wasm.send",
  "services": { "runtime": "discord-bot" },
  "config": {
    "data": {
      "type": "send_message",
      "channel_id": "{{ input.channel }}",
      "content": "{{ input.message }}"
    }
  }
}
```

If the module exports `command`, Noda calls it immediately. Otherwise, the data is buffered and delivered in the next tick's `commands` array.

**`wasm.query`** ŌĆö sync, request-response. The workflow waits for the module to return data:

```json
{
  "type": "wasm.query",
  "services": { "runtime": "game" },
  "config": {
    "data": {
      "type": "get_leaderboard",
      "limit": 10
    },
    "timeout": "5s"
  }
}
```

`wasm.query` requires a `timeout` in its config. If the module does not respond within the timeout, the node fails with a `TimeoutError`.

### 21.7 Configuration

Wasm runtimes are declared in `noda.json`:

```json
{
  "wasm_runtimes": {
    "game": {
      "module": "game.wasm",
      "tick_rate": 20,
      "encoding": "msgpack",
      "services": ["app-cache", "game-storage"],
      "connections": ["game-ws"],
      "config": {
        "max_players": 100
      }
    },
    "discord-bot": {
      "module": "discord_bot.wasm",
      "tick_rate": 10,
      "services": ["app-cache"],
      "connections": ["notifications-ws"],
      "allow_outbound": {
        "http": ["discord.com", "cdn.discordapp.com"],
        "ws": ["gateway.discord.gg"]
      },
      "config": {
        "token": "{{ $env('DISCORD_BOT_TOKEN') }}"
      }
    }
  }
}
```

Noda resolves `$env()` references in the `config` object before passing it to the module's `initialize` ŌĆö the module receives actual values, never variable references.

Each module key (e.g., `"game"`, `"discord-bot"`) auto-registers as a service instance under the `wasm` prefix. Workflow nodes reference them by this name: `"services": { "runtime": "game" }`.

### 21.8 Security Model

- **Service isolation** ŌĆö the module can only access services listed in its config. Unauthorized calls return `PERMISSION_DENIED`.
- **Network isolation** ŌĆö outbound connections only to whitelisted hosts. No wildcards.
- **Memory isolation** ŌĆö each instance has its own linear memory, enforced by Extism/Wazero.
- **No filesystem access** ŌĆö all storage goes through `noda_call` to storage services.
- **No database access** ŌĆö modules cannot access the database directly. Database operations go through workflows triggered via `noda_call("", "trigger_workflow", ...)`.
- **Call serialization** ŌĆö `tick`, `command`, and `query` are never called concurrently on the same instance.
- **Resource limits** ŌĆö Extism supports memory limits and execution timeouts per module.

### 21.9 Use Cases

- **Multiplayer game server** ŌĆö game world state lives in Wasm linear memory. Player inputs arrive via tick. State updates broadcast to clients via `noda_call("game-ws", "send", ...)`. Periodic snapshots via `noda_call("game-storage", "write", ...)`.
- **Discord / Slack / Telegram bot** ŌĆö Noda manages the gateway WebSocket connection (heartbeats, reconnection). The module receives messages per tick, processes commands, and sends replies via `noda_call_async` to the platform's HTTP REST API.
- **Trading bot** ŌĆö Noda manages the exchange WebSocket feed. The module receives price updates per tick, executes trading logic, places orders via `noda_call_async` HTTP to the exchange API. Order confirmations arrive in future ticks.
- **IoT gateway** ŌĆö aggregates sensor data from managed connections, triggers workflows for alerts, pushes real-time data to dashboards.
- **AI agent loop** ŌĆö observes, decides, and acts each tick. Fires LLM API calls via `noda_call_async`, processes responses when they arrive (possibly many ticks later). Triggers workflows for tool execution, persists memory via storage.

## 22. Node Categories

Nodes are the atomic units of workflow logic. They are organized by prefix, which maps to their providing plugin or the core engine. Every node declares its **named outputs** ŌĆö the connection ports that edges attach to.

### Core Nodes (built-in, no plugin required)

**Control flow:**

| Node | Outputs | Description |
|---|---|---|
| `control.if` | `then`, `else`, `error` | Conditional branching |
| `control.switch` | *static from config*, `default`, `error` | Multi-branch routing |
| `control.loop` | `done`, `error` | Iterate a sub-workflow over a collection |

Retries are not a node type ŌĆö they are configured on edges (see Section 8.4). Parallelism is implicit from the graph structure (see Section 8.2) ŌĆö no explicit parallel node is needed.

**Workflow**: `workflow.run` (invoke sub-workflow, outputs are dynamic from sub-workflow's `workflow.output` nodes, plus `error`; optionally wrapped in a database transaction), `workflow.output` (terminal node in sub-workflows ŌĆö declares a named output and return data)

**Transform** (all outputs: `success`, `error`): `transform.set`, `transform.map`, `transform.filter`, `transform.merge`, `transform.delete`, `transform.validate`

`transform.merge` is a **data operation** node ŌĆö it combines data from multiple upstream node outputs into a single dataset. It is not a flow control node. Flow convergence (branches reconverging after a conditional split) is handled automatically by the engine (see Section 8.2). Merge modes:

| Mode | Description |
|---|---|
| `append` | Stack all input items together into a single list |
| `match` | Combine items by matching field values (inner join, outer join, enrich) |
| `position` | Combine items by index position |

The node references upstream outputs via expressions in its config:

```json
{
  "type": "transform.merge",
  "config": {
    "mode": "match",
    "inputs": ["{{ fetch-users }}", "{{ fetch-orders }}"],
    "match": {
      "type": "inner",
      "fields": { "left": "id", "right": "user_id" }
    }
  }
}
```

For `append` and `position` modes, only `inputs` is required. For `match`, `inputs` must have exactly two entries, and `match.type` (`inner`, `outer`, `enrich`) and `match.fields` (`left`, `right`) are required.

**Response** (all outputs: `success`, `error`): `response.json`, `response.redirect`, `response.error`

**Utility** (all outputs: `success`, `error`): `util.log`, `util.uuid`, `util.delay`, `util.timestamp`

**Events** (all outputs: `success`, `error`): `event.emit` (configurable: stream or pubsub mode, optional service slots)

**Real-time** (all outputs: `success`, `error`): `ws.send`, `sse.send`

**File handling** (all outputs: `success`, `error`): `upload.handle`

**Wasm interaction** (all outputs: `success`, `error`): `wasm.send` (async, fire and forget), `wasm.query` (sync, request-response)

### Plugin Nodes (require configured service instance)

All plugin nodes have outputs `success` and `error` unless noted otherwise.

**Database** (`db` prefix): `db.query`, `db.exec`, `db.create`, `db.update`, `db.delete`

**Cache** (`cache` prefix): `cache.get`, `cache.set`, `cache.del`, `cache.exists`

**Storage** (`storage` prefix): `storage.read`, `storage.write`, `storage.delete`, `storage.list`

**Image** (`image` prefix): `image.resize`, `image.crop`, `image.watermark`, `image.convert`, `image.thumbnail`

**HTTP Client** (`http` prefix): `http.request`, `http.get`, `http.post`

**Email** (`email` prefix): `email.send`

Every plugin node **must** include service references for all declared service slots. The engine validates this at startup ŌĆö missing or mismatched references prevent the application from starting.

## 23. Workflow Testing

Noda provides a `noda test` command that runs workflow tests without requiring external services.

### 23.1 Test Definition

Tests are defined in `tests/` as JSON files following a standardized structure:

```json
{
  "id": "test-create-user",
  "workflow": "create-user",
  "tests": [
    {
      "name": "creates user successfully",
      "input": {
        "email": "test@example.com",
        "name": "Test User"
      },
      "mocks": {
        "check-duplicate": { "output": null },
        "insert-user": {
          "output": {
            "id": "uuid-123",
            "email": "test@example.com",
            "name": "Test User"
          }
        }
      },
      "expect": {
        "status": "success",
        "output": {
          "response.status": 201,
          "response.body.email": "test@example.com"
        }
      }
    },
    {
      "name": "rejects duplicate email",
      "input": {
        "email": "existing@example.com",
        "name": "Duplicate User"
      },
      "mocks": {
        "check-duplicate": {
          "output": { "id": "existing-id" }
        }
      },
      "expect": {
        "status": "success",
        "output": {
          "response.status": 409,
          "response.body.error.code": "CONFLICT"
        }
      }
    },
    {
      "name": "handles database failure",
      "input": {
        "email": "test@example.com",
        "name": "Test User"
      },
      "mocks": {
        "check-duplicate": { "output": null },
        "insert-user": {
          "error": { "code": "DB_ERROR", "message": "connection refused" }
        }
      },
      "expect": {
        "status": "error",
        "error_node": "insert-user"
      }
    }
  ]
}
```

### 23.2 How It Works

The test runner loads the workflow graph but replaces all plugin nodes with mock implementations. Each test case specifies what each mocked node should return (success with output, or error). Core nodes (control flow, transform, response) execute normally.

`workflow.run` and `control.loop` nodes can also be mocked. For `workflow.run`, the mock specifies which named output to fire and what data to return:

```json
{
  "mocks": {
    "check-inventory": {
      "output_name": "in-stock",
      "output": { "quantity": 42 }
    }
  }
}
```

If a `workflow.run` node is not mocked, the test runner loads and executes the real sub-workflow (applying mocks within the sub-workflow if specified). This allows both unit testing (mock the sub-workflow) and integration testing (run the real sub-workflow).

This means:

- No Postgres, Redis, or any external service needed to run tests
- Expression evaluation works normally against mock data
- Error paths are testable ŌĆö you can mock a node to fail and verify the error edge is followed
- The full graph is exercised ŌĆö parallel branches, retries, transactions
- Sub-workflows can be tested in isolation or as part of the parent workflow

### 23.3 CLI Integration

```
noda test                    ŌĆö run all tests
noda test --workflow create-user  ŌĆö run tests for one workflow
noda test --verbose          ŌĆö show full trace for each test
```

Test results follow a standard format with pass/fail status, expected vs actual values, and the execution trace for failed tests.

## 25. Startup Sequence

1. **Parse CLI flags** ŌĆö determine which runtimes to start (server, workers, scheduler, wasm)
2. **Load config** ŌĆö discover and parse all JSON files, apply environment overlay
3. **Validate config** ŌĆö check every file against its JSON Schema, resolve `$ref` references to shared schemas
4. **Initialize plugins** ŌĆö load plugin types, create service instances from config
5. **Register nodes** ŌĆö each plugin registers its nodes under its prefix in the service registry
6. **Validate workflows** ŌĆö check that every node type has a plugin, every service reference has an instance, every edge references existing nodes, no cycles in graphs, all `workflow.output` nodes in sub-workflows are on mutually exclusive branches
7. **Compile expressions** ŌĆö parse all `{{ }}` expressions using Expr and cache the compiled programs
8. **Build HTTP routes** ŌĆö translate route configs into Fiber handlers with middleware chains, applying group presets
9. **Build connection endpoints** ŌĆö register WebSocket/SSE paths with the connection manager
10. **Load Wasm modules** ŌĆö instantiate Extism plugins, register `noda_call` and `noda_call_async` host functions, call `initialize` exports with config and service manifests
11. **Start runtimes** ŌĆö start the HTTP server, worker consumers, scheduler, and Wasm tick loops based on CLI flags
12. **Health check** ŌĆö verify all service instances are reachable

If any step fails, the application logs clear error messages (with file paths and line references where possible) and exits without serving traffic.

## 26. Graceful Shutdown

Startup and shutdown are managed by a `Lifecycle` manager (`internal/lifecycle/`). Components register in dependency order; shutdown runs in reverse. A signal handler (SIGTERM/SIGINT) is installed early ŌĆö before runtimes start ŌĆö so a signal during startup still cleans up whatever has been started so far.

**Shutdown order** (reverse of registration):

1. **Stop HTTP server** ŌĆö stop accepting new connections, drain in-flight requests
2. **Stop workers** ŌĆö drain in-flight messages
3. **Stop scheduler** ŌĆö stop firing jobs
4. **Stop Wasm modules** ŌĆö call `shutdown()` on each module, giving them a deadline to save state
5. **Stop file watcher** ŌĆö dev mode only
6. **Close WebSocket/SSE connections** ŌĆö via the connection manager
7. **Close service connections** ŌĆö close database pools, Redis connections, storage handles
8. **Flush telemetry** ŌĆö ensure all OpenTelemetry spans and logs are exported
9. **Exit**

The shutdown deadline is configurable (default: 30 seconds). Each component gets an equal share of the total deadline. Unused budget from fast-stopping components rolls forward to the next.

**Service initialization** is deterministic: service names are sorted alphabetically before creation. Each `CreateService` call has a 30-second timeout to fail fast if external dependencies (Postgres, Redis) are unreachable.

## 27. Multi-Instance Deployment

Noda is designed to run as multiple instances behind a load balancer. The key considerations:

**HTTP Server** ŌĆö stateless. Any instance can handle any request. Load balancer distributes normally.

**Workers** ŌĆö Redis Streams consumer groups handle distribution automatically. Each instance joins the group and receives different messages. No duplicate processing.

**Scheduler** ŌĆö distributed locks via the cache service (atomic set-if-not-exists) ensure only one instance executes each scheduled job. All instances run the cron engine, but the lock prevents duplicate execution.

**Connection Manager** ŌĆö each instance holds its own set of WebSocket/SSE connections. A Redis routing table maps channels to instance IDs, enabling targeted delivery via PubSub instead of broadcast (see Section 19.5).

**Wasm Runtime** ŌĆö each Wasm module runs on a single Noda instance with its own in-memory state. For game servers, this means separate game worlds per instance. For bots, each instance can run its own bot process. Coordination between Wasm modules across instances happens through Redis events ŌĆö the same infrastructure workers use.

**Casbin** ŌĆö policy changes can be synchronized across instances via Casbin's watcher mechanism, backed by Redis PubSub.

## 32. What Noda Does NOT Do

To keep scope clear:

- **No user management** ŌĆö Noda validates JWTs and enforces policies. It does not manage user accounts, passwords, or registration flows. That is application logic built with workflows.
- **No admin UI at runtime** ŌĆö the visual editor is a development tool, not a production dashboard. Runtime monitoring uses OpenTelemetry-compatible tools (Grafana, Jaeger).
- **No frontend rendering** ŌĆö Noda serves APIs, WebSockets, and SSE. It does not serve HTML, templates, or static frontend assets beyond the OpenAPI docs UI.
- **No custom scripting runtime** ŌĆö Noda uses Expr for expressions in workflows. It does not embed Lua, JavaScript, or any Turing-complete scripting language in the workflow engine. Custom application logic runs in Wasm modules, not in workflows.
- **No in-memory fallbacks** ŌĆö if a service is required, it must be configured and reachable. Local development uses real services via the Docker Compose setup.
- **No game engine** ŌĆö the Wasm runtime hosts user-provided game engines. Noda provides infrastructure (networking, storage, auth) but contains no game logic, physics, or ECS of its own.
- **No bot framework** ŌĆö the Wasm runtime hosts user-provided bot logic. Noda provides outbound networking, storage, and event infrastructure but contains no Discord, Slack, or Telegram protocol handling of its own.
