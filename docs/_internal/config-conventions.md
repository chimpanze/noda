# Noda — Config Conventions

**Version**: 0.4.0
**Status**: Complete

This document defines the naming conventions, field patterns, and structural rules for all JSON configuration in Noda. Every node config, trigger mapping, and structure definition should follow these conventions. The goal is that once you understand the patterns, most node configs become obvious from the node name alone.

---

## 1. Expression Fields vs Static Fields

Any string value in a node's config can be a runtime expression (`{{ ... }}`) **unless** it is marked as static. Static fields are values the engine needs at startup — for graph validation, output port determination, or service slot resolution.

**Static fields** (string literals only, never expressions):
- `mode` — operation mode (`event.emit`, `transform.merge`)
- `cases` — switch branch names (`control.switch`)
- `workflow` — sub-workflow ID (`workflow.run`, `control.loop`)
- `name` — output name (`workflow.output`)
- `method` — HTTP method (`http.request`)
- `type` — match type (`transform.merge`)
- `backoff` — retry backoff strategy (edge config)

**Rule:** If the engine needs the value before execution starts, it's static. If it depends on runtime data, it's an expression.

## 2. Standard Field Names

The same concept always uses the same key across all nodes. This eliminates guesswork — if you know the concept, you know the field name.

### Primary Data

| Key | Type | Meaning | Used by |
|---|---|---|---|
| `data` | expression | Primary payload or body | `ws.send`, `sse.send`, `wasm.send`, `wasm.query` |
| `body` | expression | HTTP response/request body | `response.json`, `response.error`, `http.request` |
| `value` | expression | Value to write/set | `cache.set`, `transform.set` |
| `payload` | expression | Event payload | `event.emit` |

### Identifiers and Paths

| Key | Type | Meaning | Used by |
|---|---|---|---|
| `path` | expression | File/object path in storage | `storage.read`, `storage.write`, `storage.delete`, `upload.handle` |
| `prefix` | expression | Path prefix for listing | `storage.list` |
| `key` | expression | Cache/KV key | `cache.get`, `cache.set`, `cache.del`, `cache.exists` |
| `topic` | expression | Event topic name | `event.emit` |
| `channel` | expression | WebSocket/SSE channel | `ws.send`, `sse.send` |
| `url` | expression | Target URL | `http.request`, `response.redirect` |

### Queries and Conditions

| Key | Type | Meaning | Used by |
|---|---|---|---|
| `query` | expression | SQL query string | `db.query`, `db.exec` |
| `params` | expression[] | Parameterized query values | `db.query`, `db.exec` |
| `condition` | expression | Boolean expression for branching | `control.if` |
| `expression` | expression | Value expression for matching | `control.switch` |
| `collection` | expression | Array expression for iteration | `control.loop` |

### Sub-Workflow

| Key | Type | Meaning | Used by |
|---|---|---|---|
| `workflow` | static | Sub-workflow ID | `workflow.run`, `control.loop` |
| `input` | map of expressions | Input mapping for `$.input` | `workflow.run`, `control.loop` |
| `name` | static | Output name declaration | `workflow.output` |
| `transaction` | static bool | Wrap in database transaction | `workflow.run` |

### HTTP

| Key | Type | Meaning | Used by |
|---|---|---|---|
| `status` | expression (int) | HTTP status code | `response.json`, `response.error` |
| `headers` | map of expressions | HTTP headers | `response.json`, `http.request` |
| `cookies` | array of objects | Response cookies (each with `name`, `value`, `path`, `domain`, `max_age`, `secure`, `http_only`, `same_site`) | `response.json` |
| `method` | static | HTTP method | `http.request` |

### Control and Limits

| Key | Type | Meaning | Used by |
|---|---|---|---|
| `mode` | static | Operation mode | `event.emit`, `transform.merge` |
| `cases` | static string[] | Branch names | `control.switch` |
| `timeout` | static duration | Timeout duration | `wasm.query`, `util.delay` |
| `ttl` | expression (int) | Time-to-live in seconds | `cache.set` |
| `max_size` | static size | Size limit | `upload.handle` |
| `max_files` | static int | File count limit | `upload.handle` |
| `allowed_types` | static string[] | MIME type whitelist | `upload.handle` |

### Transform

| Key | Type | Meaning | Used by |
|---|---|---|---|
| `fields` | map of expressions | Field mappings (target: expression) | `transform.set`, `transform.merge` |
| `inputs` | expression[] | Multiple input datasets | `transform.merge` |
| `schema` | static | JSON Schema for validation | `transform.validate` |

## 3. Field Naming Rules

**Snake case.** All field names use `snake_case`: `max_size`, `allowed_types`, `content_type`. Never camelCase, never kebab-case.

**No abbreviations.** `collection`, not `coll`. `expression`, not `expr`. `condition`, not `cond`. `timeout`, not `to`. The config is machine-generated by the visual editor — brevity doesn't matter, clarity does.

**Plural for arrays.** `params`, `cases`, `inputs`, `headers`, `cookies`, `fields`, `allowed_types`. Singular for single values.

## 4. Value Formats

### Durations

Always a string with a unit suffix. Never raw numbers.

```
"5s"       — 5 seconds
"100ms"    — 100 milliseconds
"1m"       — 1 minute
"500ms"    — 500 milliseconds
"30s"      — 30 seconds
```

Supported units: `ms` (milliseconds), `s` (seconds), `m` (minutes). No hours, no days — workflows shouldn't have multi-hour timeouts.

### Sizes

Always a string with a unit suffix. Never raw bytes.

```
"10mb"     — 10 megabytes
"64kb"     — 64 kilobytes
"1gb"      — 1 gigabyte
```

Supported units: `kb`, `mb`, `gb`.

### HTTP Status Codes

Always integers: `200`, `201`, `301`, `400`, `404`, `500`. Can be expressions that resolve to integers: `{{ input.error ? 400 : 200 }}`.

## 5. Structural Patterns

### 5.1 The Sub-Workflow Pattern

Any node that invokes a sub-workflow uses the same two fields:

```json
{
  "workflow": "process-item",
  "input": {
    "item": "{{ $item }}",
    "user": "{{ auth.sub }}"
  }
}
```

`workflow` is static (the engine resolves it at startup). `input` is a map of expressions that populate the sub-workflow's `$.input`. This pattern is used by `workflow.run` and `control.loop`.

### 5.2 The Field Mapping Pattern

When a node creates or transforms data by mapping fields, it uses a flat map of `field_name: expression`:

```json
{
  "fields": {
    "full_name": "{{ input.first_name }} {{ input.last_name }}",
    "email_lower": "{{ lower(input.email) }}",
    "created_at": "{{ now() }}"
  }
}
```

Each key is the output field name, each value is an expression. Used by `transform.set`.

### 5.3 The Multi-Input Pattern

When a node combines data from multiple upstream sources, it uses `inputs` — an array of expressions:

```json
{
  "inputs": ["{{ fetch-users }}", "{{ fetch-orders }}"]
}
```

Each expression resolves to a dataset. Used by `transform.merge`.

### 5.4 The Filter/Map Expression Pattern

When a node applies a per-item expression to a collection, it uses `collection` (the array) and `expression` (applied per item). Within the expression, `$item` refers to the current element and `$index` to the current index:

```json
{
  "collection": "{{ fetch-users }}",
  "expression": "{{ $item.age >= 18 }}"
}
```

Used by `transform.filter` and `transform.map`.

## 6. Config Schema Conventions

Every node type defines a JSON Schema for its config. These schemas follow additional conventions:

**Required fields are minimal.** Only fields without sensible defaults are required. For example, `response.json` requires `status` and `body`, but `headers` and `cookies` are optional.

**Default values are documented in the schema.** If `transform.merge` mode defaults to `"append"`, the schema says so. The visual editor uses defaults to pre-fill forms.

**Enum fields list all valid values.** `mode: "stream" | "pubsub"`, `method: "GET" | "POST" | "PUT" | "PATCH" | "DELETE"`. The visual editor renders these as dropdowns.

**No nested config objects beyond one level.** A node's config should be flat or one level deep. `match.type` and `match.fields` are acceptable. `match.fields.left.nested.deep` is not. If a config needs deep nesting, the node is doing too much.

## 7. Expression Context Variables

Within expressions, the following variables are available:

### Always Available

| Variable | Type | Description |
|---|---|---|
| `input` | any | The workflow's `$.input` from trigger mapping |
| `auth` | object or nil | Authentication data (`.sub`, `.roles`, `.claims`) |
| `trigger` | object | Trigger metadata (`.type`, `.timestamp`, `.trace_id`) |

### Node Outputs

Any completed node's output is available by its node ID or `as` alias:

```
{{ fetch-user }}              — the full output
{{ fetch-user.name }}         — a field within the output
{{ fetch-user[0].email }}     — array indexing
```

### Loop Variables (inside `control.loop` input mapping only)

| Variable | Type | Description |
|---|---|---|
| `$item` | any | Current element from the collection |
| `$index` | int | Zero-based iteration index |

### Built-in Functions

Plugins can register custom functions. The core engine provides:

| Function | Description |
|---|---|
| `len(array)` | Length of an array |
| `lower(string)` | Lowercase string |
| `upper(string)` | Uppercase string |
| `now()` | Current timestamp |
| `$uuid()` | Generate a UUID |
| `$env(key)` | Read environment variable (only in root config, not in workflows) |

These are Expr library built-ins plus Noda-registered functions. The full function list is determined by the Expr library and registered plugins.

---

## 8. Edge Config

Edges have their own config fields beyond the standard `from`, `output`, `to`:

| Key | Type | Meaning |
|---|---|---|
| `retry.attempts` | static int | Max retry attempts (error edges only) |
| `retry.backoff` | static | Backoff strategy: `"fixed"`, `"exponential"` |
| `retry.delay` | static duration | Initial delay between retries |

Edge config is always static — no expressions.
